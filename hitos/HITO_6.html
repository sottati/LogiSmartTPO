1. Análisis de Candidatos a Singleton

Se identificaron recursos que deben tener una única instancia para evitar inconsistencias o sobrecostos.

Recurso Por qué único Problema con múltiples instancias
ConexionBD Punto central de acceso a persistencia Estados divergentes y auditoría fragmentada
Logger Unificar trazas del sistema Logs duplicados o con orden inconsistente
ConfiguracionApp (candidato) Parámetros globales del runtime Versiones de configuración distintas por módulo

2. Implementación de Singletons

Se implementaron dos Singletons thread-safe con double-checked locking y volatile.

2.1 ConexionBD.java

public final class ConexionBD {
  private static volatile ConexionBD instancia;
  private final List<String> historialQueries
    = new CopyOnWriteArrayList<>();

  private ConexionBD() {}

  public static ConexionBD obtenerInstancia() {
    if (instancia == null) {
      synchronized (ConexionBD.class) {
        if (instancia == null)
          instancia = new ConexionBD();
      }
    }
    return instancia;
  }
}

2.2 Logger.java

public final class Logger {
  private static volatile Logger instancia;
  private final Object lock = new Object();

  private Logger() {}

  public static Logger obtenerInstancia() {
    if (instancia == null) {
      synchronized (Logger.class) {
        if (instancia == null)
          instancia = new Logger();
      }
    }
    return instancia;
  }
}

2.3 Thread Safety

  • Constructor privado: evita instanciación con new desde el cliente.
  • volatile: previene lecturas inconsistentes entre hilos.
  • Bloque synchronized: se ejecuta únicamente en la primera inicialización.
  • Se reduce la contención de lock luego de crear la instancia.

4. Implementación de Factory Methods

Se agregaron tres fábricas en src/com/logismart/infraestructura/fabrica.

4.1 FabricaDeEnvios.java

public static Envio crearEnvio(TipoEnvio tipo, Empresa empresa,
                               LocalDateTime fecha) {
  return switch (tipo) {
    case EXPRESS   -> new EnvioExpress(UUID.randomUUID().toString(), empresa, fecha);
    case STANDARD  -> new EnvioStandard(UUID.randomUUID().toString(), empresa, fecha);
    case ECONOMICO -> new EnvioEconomico(UUID.randomUUID().toString(), empresa, fecha);
  };
}

4.2 FabricaDeNotificadores.java

public static Notificador crearNotificador(TipoNotificador tipo) {
  return switch (tipo) {
    case EMAIL -> new NotificadorEmail();
    case SMS   -> new NotificadorSMS();
    case PUSH  -> new NotificadorPush();
  };
}

4.3 FabricaDeVehiculos.java

public static Vehiculo crearVehiculo(TipoVehiculo tipo, String patente) {
  return new Vehiculo(UUID.randomUUID().toString(), patente,
                      tipo.getCapacidadKgBase(), tipo.name(), true);
}
Resultado: el controller queda desacoplado de las subclases concretas

5.4 Mejora: factories polimorficas (sin switch)

Para evitar switch en la creacion, los enums encapsulan el comportamiento: cada constante conoce como instanciar su clase concreta. Esto mantiene el Open/Closed y reduce condicionales al agregar nuevos tipos. 

// Ejemplo: el enum crea su objeto concreto
public enum TipoNotificador {
  EMAIL { public Notificador crearNotificador() { return new NotificadorEmail(); } },
  SMS   { public Notificador crearNotificador() { return new NotificadorSMS(); } },
  PUSH  { public Notificador crearNotificador() { return new NotificadorPush(); } };
}

// Factory delega en el enum
public static Notificador crearNotificador(TipoNotificador tipo) {
  return tipo.crearNotificador();
}

7. Conclusiones

  • Singleton resolvió la necesidad de recursos únicos y unificó la trazabilidad de operaciones en el sistema.
  • Factory Method eliminó el conocimiento de las subclases concretas en el controller, respetando el principio abierto/cerrado.
  • Las factories evolucionaron a un enfoque polimórfico en enums, evitando switch para crear instancias.
  • El controller mantiene su rol de orquestación y delega completamente la creación de objetos a las fábricas.
  • El diseño resultante es extensible para incorporar más patrones creacionales en entregas futuras.
