Análisis del Diseño del Hito 4

Revisión del modelo GRASP Parte I aplicado en el Hito 4. Se identifican los problemas que quedan sin resolver y se propone cómo GRASP Parte II los aborda.

1.1 Problemas identificados en el diseño del Hito 4

# Problema Ubicación Patrón que lo resuelve
P1 No existe intermediario entre la UI y el dominio. OperadorLogistico coordina operaciones que involucran múltiples clases, lo cual mezcla roles de actor y orquestador. OperadorLogistico Controller
P2 Los permisos de cada tipo de usuario no están modelados. Cualquier código que necesite verificar si un usuario puede realizar una acción debería usar instanceof, generando condicionales frágiles. Usuario y subclases Polymorphism
P3 El mecanismo de notificación no existe. TrackingService solo publica tracking pero no notifica al cliente por ningún canal. La elección de canal (Email/SMS/Push) generaría un switch o múltiples if. TrackingService, ausencia de notificador Polymorphism Indirection
P4 No hay un servicio que centralice el envío de notificaciones. La responsabilidad podría quedar en Envio o en el Controller, reduciendo su cohesión. Ausencia de clase especializada Pure Fabrication
P5 No hay persistencia de envíos. La responsabilidad de guardar/recuperar envíos no encaja ni en Envio ni en Logismart. Ausencia de repositorio Pure Fabrication
P6 La selección de ruta óptima usa un criterio fijo (distancia). Cambiar el algoritmo requeriría modificar Ruta o el Controller. Ruta.calcularDistanciaTotal() Polymorphism Protected Variations
P7 El cálculo de costo en Ruta.calcularCostoEstimado() tiene la lógica de tarifa codificada con condicionales sobre el tipo de vehículo. Agregar una nueva política rompe el método. Ruta.calcularCostoEstimado() Protected Variations
P8 La asignación de vehículos carece de política explícita. Si cambia el criterio (disponibilidad → capacidad → proximidad), hay que modificar el Controller. Ausencia de asignador Protected Variations
P9 El ETA se fija manualmente en el Controller. No existe un modelo de cálculo de tiempo que pueda variar (con/sin tráfico, por franja horaria). Ausencia de calculador de tiempo Protected Variations
P10 La exportación de reportes tiene un único formato. ReporteService necesita condicionales para soportar PDF, JSON, Excel. ReporteService Protected Variations

Aplicación del Patrón Controller

Se crea LogiSmartController como intermediario entre la UI y el dominio. Resuelve el Problema P1: ninguna clase coordinaba operaciones que involucran múltiples objetos del dominio sin mezclar roles.

2.1 Responsabilidades del Controller

Lo que SÍ hace el Controller

  • Recibe solicitudes de la UI (métodos públicos)
  • Valida permisos usando IPermisos (Polymorphism)
  • Delega validación de datos a ValidadorDeEnvios
  • Coordina colaboración entre múltiples objetos del dominio
  • Usa servicios de Pure Fabrication para notificar y persistir
  • Devuelve resultados o lanza excepciones significativas

Lo que NO hace el Controller

  • No contiene lógica de negocio propia
  • No crea directamente objetos del dominio (delega en Expert/Creator)
  • No conoce el canal de notificación (Indirection)
  • No conoce el algoritmo de selección de ruta (Protected Variations)
  • No accede a la base de datos directamente (Pure Fabrication)

2.2 Métodos principales

public class LogiSmartController {

    // Pure Fabrications inyectadas por constructor
    private final RepositorioDeEnvios repositorioEnvios;
    private final ServicioDeNotificaciones notificaciones;
    private final ValidadorDeEnvios validador;
    private final CalculadorDeRutas calculadorRutas;
    private final AsignadorDeVehiculos asignadorVehiculos;
    private final CalculadorDeCosto calculadorCosto;
    private final CalculadorDeTiempo calculadorTiempo;

    // CU-01: Crear envio
    public Envio crearEnvio(IPermisos solicitante, Empresa empresa,
                            ClienteFinal cliente, String prioridad, LocalDateTime fecha) {
        if (!solicitante.puedeCrearEnvio())                    // Polymorphism
            throw new SecurityException("Sin permiso");
        var resultado = validador.validarCreacion(...);        // Pure Fabrication
        Envio envio = new Envio(UUID.randomUUID()...);         // Creator / Expert
        repositorioEnvios.guardar(envio);                      // Pure Fabrication
        notificaciones.notificarCreacionDeEnvio(envio, cliente); // PF + Indirection
        return envio;
    }

    // CU-03: Asignar ruta
    public Ruta asignarRuta(IPermisos solicitante, String idEnvio,
                            List<Ruta> rutas, Flota flota, ClienteFinal cliente) {
        if (!solicitante.puedeAsignarRuta()) throw new SecurityException(...);
        Vehiculo v = asignadorVehiculos.asignar(flota, envio)  // Protected Variations
                         .orElseThrow(...);
        Ruta ruta = calculadorRutas.seleccionarRuta(envio, rutas); // PF + Polymorphism
        int eta = calculadorTiempo.estimarMinutos(ruta);           // Protected Variations
        envio.iniciar();                                            // Expert (Hito 4)
        notificaciones.notificarAsignacionDeRuta(envio, ruta, cliente);
        return ruta;
    }

    // CU-04: Calcular costo
    public double calcularCosto(String idEnvio) {
        return calculadorCosto.calcular(buscarEnvioOFallar(idEnvio)); // Protected Variations
    }

    // CU-05: Obtener estado | CU-06: Cancelar envio | CU-07: Ver reportes
}

Solución: Interfaz IPermisos implementada por cada tipo de usuario El Controller simplemente llama solicitante.puedeCrearEnvio() sin conocer el tipo concreto.

Aplicación de Pure Fabrication e Indirection

Se crean clases artificiales que no representan conceptos del dominio real pero son necesarias para mantener alta cohesión y bajo acoplamiento. Resuelve los Problemas P4 y P5.

4.1 Tabla de Pure Fabrications

Clase Responsabilidad Por qué no encaja en el dominio Patrón adicional
ServicioDeNotificaciones Construir mensajes contextuales y enviarlos al cliente vía canal configurado No es responsabilidad de Envio (entidad de dominio), ni del Controller (perdería cohesión) Indirection sobre Notificador
RepositorioDeEnvios Persistir, recuperar y buscar envíos por ID o estado La persistencia no es responsabilidad de Envio (violaría Expert). No encaja en Logismart (fachada del sistema) -
ValidadorDeEnvios Centralizar reglas de validación: creación de envíos y asignación de rutas Validar en el Controller lo haría un "fat controller". Validar en Envio rompe Expert (no tiene todos los datos de contexto) -
CalculadorDeRutas Orquestar la selección de ruta óptima usando la estrategia configurada No encaja en Ruta (una ruta no se elige a sí misma) ni en OperadorLogistico (pierde cohesión) Indirection sobre CalculadorDeRuta

4.2 Indirection: ServicioDeNotificaciones como intermediario

GRASP Indirection: ServicioDeNotificaciones actúa como intermediario entre el Controller y la implementación concreta del canal. El Controller no sabe si se usa Email, SMS o Push - solo le pide al servicio que notifique. 
// ServicioDeNotificaciones depende de la interfaz Notificador (no de la clase concreta)
public class ServicioDeNotificaciones {
    private final Notificador notificador;  // inyectado: Email, SMS o Push

    public void notificarCreacionDeEnvio(Envio envio, ClienteFinal cliente) {
        String asunto = "Envio creado - ID: " + envio.getId();
        String mensaje = String.format("Hola %s, tu envio %s fue registrado...",
                cliente.getNombre(), envio.getId());
        notificador.notificar(cliente.getEmail(), asunto, mensaje);
        // El Controller NO sabe si esto envia un Email, SMS o Push
    }
}

4.3 CalculadorDeRutas como fachada sobre la estrategia

// CalculadorDeRutas encapsula el punto de variacion y permite cambiar estrategia en runtime
public class CalculadorDeRutas {
    private CalculadorDeRuta estrategia;  // RutaMasCercana, RutaMasBarata, etc.

    public Ruta seleccionarRuta(Envio envio, List<Ruta> rutas) {
        return estrategia.calcular(envio, rutas);
    }

    // Cambio de estrategia en tiempo de ejecucion (Protected Variations)
    public void cambiarEstrategia(CalculadorDeRuta nueva) {
        this.estrategia = nueva;
    }
}

Aplicación del Patrón Protected Variations

Se identifican los puntos de variación más probables y se los protege con interfaces. Resuelve los Problemas P7, P8, P9 y P10.

5.1 Puntos de variación identificados

CalculadorDeCosto Política de tarifa del envío CalculadorDeCostoFijo, CalculadorDeCostoPorPeso, CalculadorDeCostoPorDistancia Tarifa fija → por peso → por distancia → combinada (negocio cambia cada temporada)

5.2 CalculadorDeCosto - Ejemplo completo

// Protected Variations: la politica de costo es un punto de variacion
interface CalculadorDeCosto {
    double calcular(Envio envio);
}

class CalculadorDeCostoFijo implements CalculadorDeCosto {
    private final double tarifaFija;
    public double calcular(Envio envio) { return tarifaFija; }
}

class CalculadorDeCostoPorPeso implements CalculadorDeCosto {
    private final double costoPorBulto;
    public double calcular(Envio envio) {
        int bultos = envio.getOrdenes().stream().mapToInt(Orden::getTotalBultos).sum();
        return bultos * costoPorBulto;
    }
}

class CalculadorDeCostoPorDistancia implements CalculadorDeCosto {
    private final double costoPorKm;
    public double calcular(Envio envio) { return ruta.calcularDistanciaTotal() * costoPorKm; }
}

// En el Controller, nunca cambia, sin importar la politica:
double costo = calculadorCosto.calcular(envio);  // OCP: abierto a extension, cerrado a modificacion

Actividad 6 - Documentación de Decisiones de Diseño

Para cada patrón GRASP Parte II aplicado se documenta: dónde se aplicó, qué problema resolvió, cómo mejora el diseño y qué interfaces o clases se crearon.

6.1 Controller

PreguntaRespuesta
¿Dónde lo aplicaste? En la clase LogiSmartController, ubicada en com.logismart.aplicacion. Es el único punto de entrada desde la UI para los casos de uso principales del sistema.
¿Cuál fue el problema que resolvió? OperadorLogistico concentraba tanto el rol de actor del dominio como el de orquestador de operaciones multi-clase (crear envíos, asignar rutas, gestionar flota). Esto mezclaba responsabilidades de capa de presentación con lógica de dominio, violando la separación de capas.
¿Cómo mejora el diseño?
  • La UI tiene un único punto de entrada con métodos claros por caso de uso.
  • Las entidades del dominio (Envio, Ruta, Vehiculo) no saben de la UI.
  • El Controller delega toda la lógica: permisos a IPermisos, validación a ValidadorDeEnvios, persistencia a RepositorioDeEnvios.
  • Cambiar la interfaz gráfica (web → mobile → CLI) no requiere tocar el dominio.
¿Qué interfaces o clases creaste? Nueva clase LogiSmartController - con métodos: crearEnvio(), agregarOrden(), asignarRuta(), calcularCosto(), obtenerEstado(), cancelarEnvio(), verificarPermisoReporte().

6.2 Polymorphism

PreguntaRespuesta
¿Dónde lo aplicaste? En tres puntos de variación:
  1. Permisos de usuario: interfaz IPermisos implementada por las 5 subclases de Usuario.
  2. Canal de notificación: interfaz Notificador con implementaciones Email, SMS y Push.
  3. Estrategia de selección de ruta: interfaz CalculadorDeRuta con tres criterios distintos.
¿Cuál fue el problema que resolvió? Sin interfaces, el código que verificaba permisos o elegía canal usaría instanceof o switch: frágiles ante nuevos tipos. Por ejemplo: if (u instanceof OperadorLogistico) { ... } else if (u instanceof AdminEmpresa) { ... }. Cada nuevo rol o canal obligaría a modificar el Controller.
¿Cómo mejora el diseño?
  • Agregar un nuevo tipo de usuario = nueva clase que implementa IPermisos. El Controller no cambia.
  • Agregar un canal (WhatsApp) = nueva clase que implementa Notificador. ServicioDeNotificaciones no cambia.
  • Agregar un criterio de ruta = nueva clase que implementa CalculadorDeRuta. El Controller no cambia.
  • Cumple el Open/Closed Principle: abierto a extensión, cerrado a modificación.
¿Qué interfaces o clases creaste? 3 interfaces nuevas IPermisos · Notificador · CalculadorDeRuta

8 implementaciones NotificadorEmail · NotificadorSMS · NotificadorPush · RutaMasCercana · RutaMenosCongestionada · RutaMasBarata
5 clases modificadas AdminEmpresa · OperadorLogistico · Transportista · ClienteFinal · AdminPlataforma (implementan IPermisos)

6.3 Pure Fabrication

PreguntaRespuesta
¿Dónde lo aplicaste? En cuatro clases artificiales de la capa com.logismart.aplicacion, creadas para resolver responsabilidades que no encajan en ninguna entidad del dominio: ServicioDeNotificaciones, RepositorioDeEnvios, ValidadorDeEnvios y CalculadorDeRutas.
¿Cuál fue el problema que resolvió? Tres responsabilidades no tenían hogar natural en el dominio:
  • Enviar notificaciones: no es responsabilidad de Envio ni del Controller.
  • Persistir envíos: una entidad no debería saber cómo guardarse a sí misma.
  • Validar datos de entrada: el Controller se volvería un "fat controller" si lo hiciera él solo.
  • Seleccionar ruta óptima: ni Ruta se elige a sí misma, ni el Controller debe conocer el algoritmo.
¿Cómo mejora el diseño?
  • Las entidades del dominio (Envio, Ruta) mantienen alta cohesión: solo modelan el negocio.
  • El Controller permanece delgado: delega en los servicios especializados.
  • Cada Pure Fabrication tiene una sola responsabilidad, facilitando el testing y el reemplazo independiente.
  • RepositorioDeEnvios puede conectarse a una BD real sin cambiar el Controller ni el dominio.
¿Qué interfaces o clases creaste? 4 clases nuevas ServicioDeNotificaciones (capa aplicacion) · RepositorioDeEnvios (capa aplicacion) · ValidadorDeEnvios (capa aplicacion) · CalculadorDeRutas (capa aplicacion - también actúa como Indirection)

6.4 Indirection

PreguntaRespuesta
¿Dónde lo aplicaste? En dos cadenas de delegación:
  1. ServicioDeNotificaciones actúa como intermediario entre el Controller y la interfaz Notificador (Email, SMS, Push).
  2. CalculadorDeRutas actúa como intermediario entre el Controller y la interfaz CalculadorDeRuta (las 3 estrategias).
¿Cuál fue el problema que resolvió? Sin el intermediario, el Controller dependería directamente de NotificadorEmail o de RutaMasCercana. Cambiar el canal o el algoritmo de ruta requeriría modificar el Controller (alta dependencia directa = alto acoplamiento).
¿Cómo mejora el diseño?
  • El Controller solo conoce la interfaz abstracta, nunca la implementación concreta.
  • Cambiar de Email a SMS = cambiar el Notificador inyectado. El Controller y ServicioDeNotificaciones no se modifican.
  • Cambiar la estrategia de ruta = llamar a calculadorRutas.cambiarEstrategia(nueva). El Controller no cambia.
  • Bajo acoplamiento entre capas: la capa de aplicación no conoce detalles de la capa de infraestructura.
¿Qué interfaces o clases creaste? Los intermediarios ya detallados en Pure Fabrication: ServicioDeNotificaciones y CalculadorDeRutas. Las interfaces que abstraen el destino: 2 interfaces Notificador · CalculadorDeRuta (también clasificadas bajo Polymorphism, dado que los dos patrones se aplican juntos en estos puntos).

6.5 Protected Variations

PreguntaRespuesta
¿Dónde lo aplicaste? En cuatro puntos de variación de la capa com.logismart.infraestructura:
  • Política de tarifa: interfaz CalculadorDeCosto (paquete costo).
  • Asignación de vehículo: interfaz AsignadorDeVehiculos (paquete vehiculo).
  • Estimación de tiempo de entrega: interfaz CalculadorDeTiempo (paquete tiempo).
  • Formato de exportación de reportes: interfaz GeneradorDeReportes (paquete reporte).
¿Cuál fue el problema que resolvió? Los algoritmos de costo, asignación de vehículo, cálculo de tiempo y exportación de reportes eran puntos de variación altamente probables: el negocio puede cambiar la política de tarifas por temporada, agregar soporte de tráfico real al ETA, o exportar reportes en un nuevo formato. Sin abstracción, cada cambio requería modificar el Controller o las entidades del dominio.
¿Cómo mejora el diseño?
  • Cada variación está encapsulada detrás de su interfaz. El Controller llama a calcular(), asignar(), estimarMinutos() o exportar() sin saber la implementación.
  • Agregar una nueva política = nueva clase que implementa la interfaz. Zero modificaciones al Controller.
  • Las implementaciones son intercambiables en runtime via inyección de dependencias (constructor del Controller).
  • Cumple el Open/Closed Principle en los cuatro puntos de variación.
¿Qué interfaces o clases creaste? 4 interfaces nuevas CalculadorDeCosto · AsignadorDeVehiculos · CalculadorDeTiempo · GeneradorDeReportes

10 implementaciones nuevas CalculadorDeCostoFijo · CalculadorDeCostoPorPeso · CalculadorDeCostoPorDistancia · AsignadorPorDisponibilidad · AsignadorPorCapacidad · CalculadorDeTiempoSimple · CalculadorDeTiempoConTrafico · GeneradorDeReportesPDF · GeneradorDeReportesJSON · RutaMasBarata (también cubre Protected Variations sobre el criterio de ruta)

Conclusiones y Justificación de Decisiones

Síntesis de cómo los cinco patrones trabajan en conjunto y qué beneficios concretos aporta el nuevo diseño respecto al Hito 4.

7.1 Cómo cambió el diseño

PatrónDónde se aplicóProblema resueltoBeneficio concreto
Controller LogiSmartController OperadorLogistico mezclaba rol de actor y orquestador La UI tiene un punto de entrada único; el dominio no sabe de la UI
Polymorphism IPermisos, Notificador, CalculadorDeRuta Condicionales instanceof y switch dispersos Agregar un nuevo tipo de usuario, canal o estrategia = agregar clase, no modificar existente
Pure Fabrication ServicioDeNotificaciones, RepositorioDeEnvios, ValidadorDeEnvios, CalculadorDeRutas Responsabilidades técnicas contaminando entidades del dominio Las entidades del dominio permanecen cohesivas; la infraestructura está aislada
Indirection ServicioDeNotificaciones → Notificador, CalculadorDeRutas → CalculadorDeRuta Acoplamiento directo entre servicios de aplicación e implementaciones concretas Cambiar de Email a SMS no requiere tocar el Controller ni ServicioDeNotificaciones
Protected Variations CalculadorDeCosto, AsignadorDeVehiculos, CalculadorDeTiempo, GeneradorDeReportes Algoritmos y políticas codificados en el Controller o en entidades del dominio Cambiar una política = nueva implementación de la interfaz, sin tocar el Controller

7.2 Métricas de mejora del diseño

Hito 4 - Situación de partida

  • Sin Controller: la UI accedía directamente al dominio
  • Sin permisos polimórficos: riesgo de instanceof en cada verificación
  • Sin servicio de notificaciones: canal hardcodeado en TrackingService
  • Sin persistencia: los envíos no sobrevivían entre llamadas
  • Algoritmos fijos: costo, asignación y tiempo de entrega sin abstracción

Hito 5 - Diseño refactorizado

  • 1 Controller coordina todas las operaciones principales
  • 3 interfaces de Polymorphism: IPermisos, Notificador, CalculadorDeRuta
  • 4 Pure Fabrications: SrvNotificaciones, Repositorio, Validador, CalcRutas
  • 4 interfaces de Protected Variations con 10 implementaciones
  • 0 condicionales instanceof en el Controller

7.3 Integración de los cinco patrones

Los patrones trabajan en cadena:
  1. La UI llama al Controller, que es el único punto de entrada.
  2. El Controller verifica permisos usando Polymorphism (IPermisos), sin instanceof.
  3. El Controller delega en Pure Fabrications (validar, persistir, notificar, seleccionar ruta).
  4. Las Pure Fabrications usan Indirection para hablar con implementaciones concretas a través de interfaces.
  5. Los algoritmos variables se protegen con Protected Variations: el Controller invoca interfaces, no clases concretas.

7.4 Beneficios concretos del nuevo diseño

✅ Mantenibilidad

  • Cambios localizados: nueva política = nueva clase, cero modificación al Controller
  • Entidades del dominio cohesivas: no mezclan lógica de negocio con infraestructura
  • Claridad de responsabilidades: cada clase sabe qué hace sin ambigüedad

✅ Testabilidad

  • Inyección de dependencias: fácil crear mocks de interfaces
  • Pruebas del Controller sin tocar persistencia, notificaciones ni algoritmos reales
  • Cada clase testeable en aislamiento

✅ Escalabilidad

  • Agregar nuevos canales de notificación sin modificar nada existente
  • Nuevas políticas de costo, asignación o tiempo = nuevas implementations
  • El sistema crece sin explosión de condicionales

✅ Bajo acoplamiento

  • UI no conoce el dominio directamente: MediaPlayer via Controller
  • Implementaciones concretas desconocidas en capas superiores
  • Máxima flexibilidad para cambiar detalles de implementación

7.5 Próximos pasos

Hacia el Hito 6 - Patrones Creacionales:
  1. Singleton para recursos únicos: Aplicar a AdminPlataforma (un único administrador del sistema) y potencialmente a ConexionBD si se implementa persistencia.
  2. Factory Method para creación desacoplada: Crear VehiculoFactory, EnvioFactory y NotificadorFactory para instanciar objetos sin acoplar el cliente.
  3. Abstract Factory para familias regionales: Si LogiSmart expande a múltiples regiones, crear VehiculoFactoryArgentina, VehiculoFactoryBrasil, etc.
  4. Builder para objetos complejos: Implementar EnvioBuilder para construir Envio con múltiples atributos opcionales de forma legible.
  5. Prototype para clonación eficiente: Si se necesita duplicar envíos con la misma configuración en lote.
⚠️ Validaciones antes de Hito 6:
  • Verificar que el Controller no tiene lógica de negocios (solo orquesta).
  • Confirmar que cada interfaz de polymorphism tiene al menos 2 implementaciones.
  • Revisar que las Pure Fabrications no dependen una de otra (grafo de dependencias acíclico).
  • Documentar qué variaciones están protegidas y por qué.
