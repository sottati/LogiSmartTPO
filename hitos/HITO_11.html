Contexto e integración

El Hito 11 completa el catálogo GoF de patrones de comportamiento incorporando Iterator, Mediator, Memento y Observer al sistema LogiSmart. Estos cuatro patrones abordan la travesía uniforme de colecciones, la coordinación desacoplada entre componentes, la gestión de historial de estados y la notificación reactiva a cambios.

Actividad 1 · Iterator

Identificación de candidatos

Se analizaron los contextos del sistema donde múltiples elementos deben recorrerse de forma uniforme, independientemente de cómo estén almacenados internamente.

GoF · Iterator

Candidatos identificados

CandidatoEstructura internaProblema sin IteratorPor qué Iterator resuelve
Repositorio de envíos activosArray de tamaño fijo (O(1) por índice)El cliente debe conocer la capacidad máxima y el índice actual.El iterador encapsula el cursor; el cliente solo llama tieneSiguiente() / obtenerSiguiente().
Cola de envíos pendientesLista enlazada (inserción/remoción O(1) en cabeza)El recorrido requiere navegar punteros Nodo.siguiente manualmente.El iterador de lista oculta la estructura enlazada detrás de la misma interfaz.
Índice de envíos por IDHashMap<String, Envio> (lookup O(1))Iterar un HashMap requiere obtener values().iterator(), acoplando al cliente a la implementación Java.El iterador adapta el java.util.Iterator interno a la interfaz propia del dominio.

Diseño de la jerarquía

Se definieron dos interfaces en el paquete infraestructura.comportamiento.iterator: IteradorEnvios (el cursor) y ColeccionEnvios (la fábrica del cursor). Cada colección concreta crea su propio iterador como clase interna, garantizando acceso al estado privado sin exponerlo al exterior.

ClaseRol GoFEstructura internaComplejidad de iteración
ColeccionArrayConcreteAggregateEnvio[1000] + int tamañoO(n) - acceso directo por índice
ColeccionListaConcreteAggregateNodos enlazados (Nodo.siguiente)O(n) - seguimiento de punteros
ColeccionHashConcreteAggregateHashMap<String,Envio>O(n) - orden no garantizado
IteradorEnviosIterator (interfaz)-tieneSiguiente(), obtenerSiguiente(), reiniciar()
Decisión de diseño: Los iteradores son clases internas (inner classes) de cada colección concreta. Esto les permite acceder a los campos privados (envios[], cabeza, map) sin getters adicionales, manteniendo el encapsulamiento de la estructura de datos.
Actividad 2 · Iterator

Implementación

Se implementaron las dos interfaces y tres colecciones concretas, cada una con su iterador interno. El cliente (SistemaLogisticaEventDriven) usa únicamente la interfaz ColeccionEnvios y nunca accede a la estructura interna.

GoF · Iterator
// Interfaz del iterador
public interface IteradorEnvios {
    boolean tieneSiguiente();
    Envio   obtenerSiguiente();
    void    reiniciar();
}

// Interfaz de la colección
public interface ColeccionEnvios {
    IteradorEnvios crearIterador();
    void agregar(Envio envio);
    void remover(Envio envio);
    int  obtenerTamaño();
}
// Colección basada en array
public class ColeccionArray implements ColeccionEnvios {
    private static final int CAPACIDAD_MAX = 1000;
    private final Envio[] envios = new Envio[CAPACIDAD_MAX];
    private int tamaño = 0;

    public IteradorEnvios crearIterador() {
        return new IteradorArray();
    }
    public void agregar(Envio e) {
        if (tamaño < CAPACIDAD_MAX) envios[tamaño++] = e;
    }
    public void remover(Envio e) {
        for (int i = 0; i < tamaño; i++) {
            if (envios[i].getId().equals(e.getId())) {
                System.arraycopy(envios, i+1, envios, i, tamaño-i-1);
                envios[--tamaño] = null; return;
            }
        }
    }
    public int obtenerTamaño() { return tamaño; }

    private class IteradorArray implements IteradorEnvios {
        private int indice = 0;
        public boolean tieneSiguiente() { return indice < tamaño; }
        public Envio   obtenerSiguiente() { return envios[indice++]; }
        public void    reiniciar()        { indice = 0; }
    }
}
// Colección basada en lista enlazada
public class ColeccionLista implements ColeccionEnvios {
    private static class Nodo {
        Envio dato; Nodo siguiente;
        Nodo(Envio e) { this.dato = e; }
    }
    private Nodo cabeza; private int tamaño = 0;

    public void agregar(Envio e) { Nodo n = new Nodo(e); n.siguiente = cabeza; cabeza = n; tamaño++; }
    public void remover(Envio e) { /* maneja cabeza y nodos intermedios */ }
    public int  obtenerTamaño()  { return tamaño; }
    public IteradorEnvios crearIterador() { return new IteradorLista(); }

    private class IteradorLista implements IteradorEnvios {
        private Nodo actual = cabeza;
        public boolean tieneSiguiente() { return actual != null; }
        public Envio   obtenerSiguiente() { Envio e = actual.dato; actual = actual.siguiente; return e; }
        public void    reiniciar()        { actual = cabeza; }
    }
}

// Colección basada en HashMap
public class ColeccionHash implements ColeccionEnvios {
    private final Map<String, Envio> mapa = new HashMap<>();
    public void agregar(Envio e)  { mapa.put(e.getId(), e); }
    public void remover(Envio e)  { mapa.remove(e.getId()); }
    public int  obtenerTamaño()   { return mapa.size(); }
    public IteradorEnvios crearIterador() { return new IteradorHash(); }

    private class IteradorHash implements IteradorEnvios {
        private Iterator<Envio> interno = mapa.values().iterator();
        public boolean tieneSiguiente() { return interno.hasNext(); }
        public Envio   obtenerSiguiente() { return interno.next(); }
        public void    reiniciar()        { interno = mapa.values().iterator(); }
    }
}

Diseño del pipeline de eventos

Se modeló el procesamiento de un envío como un pipeline de eventos: cada componente finaliza su tarea notificando un evento al mediador, quien decide qué componente activa a continuación. Esto convierte el flujo en event-driven sin acoplamiento directo.

EventoEmisorAcciones del MediatorResultado
ENVIO_CREADOCentroDistribucionMediatorAudita + llama a ValidadorEnvioMediator.validar()Inicio del pipeline
VALIDACION_OKValidadorEnvioMediatorAudita + llama a SistemaPago.procesarPago()Envío aprobado, procede al pago
VALIDACION_FALLIDAValidadorEnvioMediatorAudita (fin del pipeline)Envío rechazado sin procesar pago
PAGO_CONFIRMADOSistemaPagoAudita + llama a SistemaNotificacion.enviarConfirmacion()Pago ok, notifica al cliente
NOTIFICACION_ENVIADASistemaNotificacionAudita + llama a CentroDistribucionMediator.registrarEnvio()Cierre del flujo comercial
ENVIO_REGISTRADOCentroDistribucionMediatorAudita (fin del pipeline)Pipeline completo
Ventaja clave: ValidadorEnvioMediator no tiene ninguna referencia a SistemaPago. Puede cambiarse el orden del pipeline, agregar un paso o eliminar uno modificando únicamente MediadorEnviosConcreto.notificar(), sin tocar ningún componente existente.
Actividad 4 · Mediator

Implementación

Se implementaron la interfaz MediadorEnvios, el mediador concreto MediadorEnviosConcreto, cinco componentes colaboradores (CentroDistribucionMediator, ValidadorEnvioMediator, SistemaPago, SistemaNotificacion, SistemaAuditoria) con capacidad de consulta por tipo de evento.

GoF · Mediator
// Interfaz del Mediator
public interface MediadorEnvios {
    void registrarCentro(CentroDistribucionMediator c);
    void registrarValidador(ValidadorEnvioMediator v);
    void registrarPago(SistemaPago p);
    void registrarNotificador(SistemaNotificacion n);
    void registrarAuditoria(SistemaAuditoria a);
    void notificar(String evento, Object datos);
}
// Componente concreto - cada uno solo conoce al mediador
public class ValidadorEnvioMediator {
    private final MediadorEnvios mediador;

    public ValidadorEnvioMediator(MediadorEnvios mediador) {
        this.mediador = mediador;
    }

    public void validar(Envio envio) {
        boolean valido = envio.getCosto() > 0
                      && envio.getPeso()  > 0;
        String evento = valido
            ? "VALIDACION_OK" : "VALIDACION_FALLIDA";
        mediador.notificar(evento, envio);
    }
}
// Mediador concreto - orquesta el pipeline completo
public class MediadorEnviosConcreto implements MediadorEnvios {
    private CentroDistribucionMediator  centro;
    private ValidadorEnvioMediator      validador;
    private SistemaPago         pago;
    private SistemaNotificacion notificador;
    private SistemaAuditoria    auditoria;

    // registrar* - setters simples (omitidos por brevedad)

    public void notificar(String evento, Object datos) {
        switch (evento) {
            case "ENVIO_CREADO":
                auditoria.registrar("ENVIO_CREADO", datos);
                validador.validar((Envio) datos);
                break;
            case "VALIDACION_OK":
                auditoria.registrar("VALIDACION_OK", datos);
                pago.procesarPago((Envio) datos);
                break;
            case "VALIDACION_FALLIDA":
                auditoria.registrar("VALIDACION_FALLIDA", datos);
                break;                       // ← pipeline se detiene aquí
            case "PAGO_CONFIRMADO":
                auditoria.registrar("PAGO_CONFIRMADO", datos);
                notificador.enviarConfirmacion((Envio) datos);
                break;
            case "NOTIFICACION_ENVIADA":
                auditoria.registrar("NOTIFICACION_ENVIADA", datos);
                centro.registrarEnvio((Envio) datos);
                break;
            case "ENVIO_REGISTRADO":
                auditoria.registrar("ENVIO_REGISTRADO", datos);
                break;                       // ← fin del pipeline exitoso
        }
    }
}

Roles del patrón

Rol GoFClase en LogiSmartPaqueteResponsabilidad
OriginatorEnviodominioCrea el snapshot con crearMemento() y se restaura con restaurarDesdeMemento(MementoEnvio).
MementoMementoEnviodominioCaptura inmutable de (estado, origen, destino, peso, costo, timestamp). No tiene setters.

Decisión de diseño - navegación bidireccional

En lugar de un stack LIFO (solo deshacer el último estado), se eligió navegación bidireccional con índice entero: irAlEstadoAnterior(), irAlEstadoSiguiente() e irAlEstado(int). Esta decisión responde a la necesidad de auditoría logística: un operador debe poder revisar cualquier punto del ciclo de vida de un envío, no solo el inmediatamente anterior.

Actividad 6 · Memento

Implementación

Se implementaron MementoEnvio (snapshot inmutable), los métodos crearMemento() y restaurarDesdeMemento() en Envio, y el cuidador HistorialEnvios con navegación bidireccional y descarte de estados futuros al guardar desde una posición intermedia.

GoF · Memento
// Memento - snapshot inmutable en dominio
public class MementoEnvio {
    private final String estado, origen, destino;
    private final double peso, costo;
    private final long   timestamp;

    public MementoEnvio(String estado, String origen,
            String destino, double peso, double costo) {
        this.estado    = estado;
        this.origen    = origen;
        this.destino   = destino;
        this.peso      = peso;
        this.costo     = costo;
        this.timestamp = System.currentTimeMillis();
    }
    // solo getters - no hay setters
    public String obtenerEstado()    { return estado; }
    public String obtenerOrigen()    { return origen; }
    public double obtenerPeso()      { return peso;   }
    public long   obtenerTimestamp() { return timestamp; }
}
// Originator - métodos en Envio
public MementoEnvio crearMemento() {
    return new MementoEnvio(
        estado, origen, destino, peso, costo
    );
}

public void restaurarDesdeMemento(MementoEnvio m) {
    this.estado  = m.obtenerEstado();
    this.origen  = m.obtenerOrigen();
    this.destino = m.obtenerDestino();
    this.peso    = m.obtenerPeso();
    this.costo   = m.obtenerCosto();
}

public void cambiarEstado(String nuevoEstado) {
    this.estado = nuevoEstado;
    notificarObservadores(); // integración con Observer
}
// Caretaker - HistorialEnvios con navegación bidireccional
public class HistorialEnvios {
    private final List<MementoEnvio> historial    = new ArrayList<>();
    private int                      indiceActual = -1;

    public void guardarEstado(Envio envio) {
        // Si navegamos hacia atrás y guardamos → descartar estados futuros
        while (indiceActual < historial.size() - 1)
            historial.remove(historial.size() - 1);
        historial.add(envio.crearMemento());
        indiceActual++;
    }

    public void irAlEstadoAnterior(Envio envio) {
        if (indiceActual > 0)
            envio.restaurarDesdeMemento(historial.get(--indiceActual));
    }

    public void irAlEstadoSiguiente(Envio envio) {
        if (indiceActual < historial.size() - 1)
            envio.restaurarDesdeMemento(historial.get(++indiceActual));
    }

    public void irAlEstado(int indice, Envio envio) {
        if (indice >= 0 && indice < historial.size()) {
            indiceActual = indice;
            envio.restaurarDesdeMemento(historial.get(indiceActual));
        }
    }
    public int obtenerTamaño()      { return historial.size(); }
    public int obtenerIndiceActual() { return indiceActual;    }
}

// Interfaz funcional - en paquete dominio
@FunctionalInterface  // opcional; ya lo es por tener un método
public interface ObservadorEnvio {
    void actualizar(Envio envio);
}
// Subject - fragmento de Envio.java
private final List<ObservadorEnvio> observadores
    = new ArrayList<>();

public void adjuntarObservador(ObservadorEnvio o) {
    observadores.add(o);
}
public void desadjuntarObservador(ObservadorEnvio o) {
    observadores.remove(o);
}
private void notificarObservadores() {
    for (ObservadorEnvio o : observadores)
        o.actualizar(this);
}
public void cambiarEstado(String nuevoEstado) {
    this.estado = nuevoEstado;
    notificarObservadores();
}
public void iniciar() {
    this.estado = "EN_CURSO";
    notificarObservadores();
}
public void cancelar() {
    this.estado = "CANCELADO";
    notificarObservadores();
}
// Observador concreto - SistemaAuditoria
public class SistemaAuditoria implements ObservadorEnvio {
    public void actualizar(Envio envio) {
        System.out.println("[Auditoria-Obs] LOG: envio=" + envio.getId()
            + " | estado=" + envio.obtenerEstado()
            + " | ts=" + System.currentTimeMillis());
    }
}

🔀 Análisis de trade-off arquitectónico: ubicación de ObservadorEnvio y MementoEnvio

Al implementar los patrones Observer y Memento surgió una tensión real entre dos principios arquitectónicos. Esta sección documenta el análisis, las opciones evaluadas y la decisión tomada.

El problema concreto: Envio (clase de dominio, Originator/Subject) necesita referenciar MementoEnvio y ObservadorEnvio. Si estas clases viven en infraestructura.comportamiento.*, el paquete dominio dependería de infraestructura, violando el principio de capas limpias.

Opciones evaluadas

OpciónUbicaciónVentajasDesventajas
A - Cohesión de patrón infraestructura.comportamiento.observer y infraestructura.comportamiento.memento Todos los elementos de un patrón quedan en el mismo paquete. Fácil de navegar siguiendo la estructura GoF. Viola arquitectura limpia: dominio.Envio importa de infraestructura. La capa de dominio ya no es autónoma; no puede compilarse ni probarse sin infraestructura.
B - DIP en dominio ✅ Elegida dominio El dominio define el contrato (interfaz/snapshot), la infraestructura lo implementa. El dominio compila y se prueba de forma aislada. Sigue Ports & Adapters / Hexagonal Architecture. La interfaz ObservadorEnvio y la clase MementoEnvio no acompañan físicamente a los demás elementos del patrón GoF; requiere que el desarrollador busque en dos paquetes.
C - Clase interna Envio como clase interna Máxima cohesión: el Memento vive dentro del Originator, como propone GoF originalmente. Envio.java se vuelve muy grande. La interfaz ObservadorEnvio debería ser pública para usarla desde infraestructura, lo que la hace un detalle de Envio expuesto hacia afuera - contraproducente.

Decisión adoptada: Opción B

Se colocaron tanto ObservadorEnvio como MementoEnvio en com.logismart.dominio por las siguientes razones:

  • Principio de Inversión de Dependencias (DIP): el dominio define contratos (ports), la infraestructura aporta implementaciones (adapters). Los observadores concretos en infraestructura implementan ObservadorEnvio del dominio - la flecha de dependencia apunta hacia adentro, no hacia afuera.
  • Autonomía del dominio: el paquete dominio no importa nada de infraestructura. Puede compilarse y testearse de forma completamente independiente.
  • Naturaleza de los artefactos: MementoEnvio es un snapshot de datos puramente de dominio (estado del ciclo de vida, atributos logísticos). No es un mecanismo técnico genérico; es una "fotografía" de una entidad de negocio concreta. Pertenece en dominio por contenido, no solo por conveniencia técnica.
  • Precedente en el proyecto: esta arquitectura ya se aplica en Hitos 8–10 (interfaces de dominio, implementaciones en infraestructura).
Compensación adoptada para mitigar la desventaja: Los JavaDoc de MementoEnvio y ObservadorEnvio documentan explícitamente el trade-off y el razonamiento. El desarrollador que llegue a esos archivos encontrará la explicación sin necesidad de buscarla en otro lugar.

Actividad 10 · Integración Event-Driven

SistemaLogisticaEventDriven - los 4 patrones colaboran

SistemaLogisticaEventDriven orquesta los cuatro patrones en un único pipeline. Cada envío pasa por las cuatro capas de patrón en secuencia: Memento guarda el estado antes, Iterator incorpora el envío a la colección, Observer suscribe reaccionantes, y Mediator dispara el pipeline de procesamiento.

// Constructor - construye el pipeline completo
public SistemaLogisticaEventDriven() {
    // Mediator
    this.mediador   = new MediadorEnviosConcreto();
    this.centro     = new CentroDistribucionMediator(mediador);
    ValidadorEnvioMediator      val  = new ValidadorEnvioMediator(mediador);
    SistemaPago         pag  = new SistemaPago(mediador);
    SistemaNotificacion not  = new SistemaNotificacion(mediador);
    this.auditoria  = new SistemaAuditoria();
    mediador.registrarCentro(centro);
    mediador.registrarValidador(val);
    mediador.registrarPago(pag);
    mediador.registrarNotificador(not);
    mediador.registrarAuditoria(auditoria);
    // Iterator
    this.coleccion = new ColeccionArray();
    // Memento
    this.historial = new HistorialEnvios();
}
// procesarEnvios - activa los 4 patrones
public void procesarEnvios(List<Envio> envios) {
    ObservadorEnvio centroObs    = new CentroDistribucionObservador();
    ObservadorEnvio notifObs     = new SistemaNotificacionObservador();
    ObservadorEnvio auditoriaObs = new SistemaAuditoria();

    for (Envio envio : envios) {
        // 1. Memento - snapshot antes del pipeline
        historial.guardarEstado(envio);
        // 2. Iterator - ingresa a la colección
        coleccion.agregar(envio);
        // 3. Observer - suscribir reactivos
        envio.adjuntarObservador(centroObs);
        envio.adjuntarObservador(notifObs);
        envio.adjuntarObservador(auditoriaObs);
        // 4. Mediator - dispara el pipeline
        centro.crearEnvio(envio);
        // 5. Memento - snapshot después del pipeline
        historial.guardarEstado(envio);
    }
}

Conclusión

El Hito 11 incorpora cuatro patrones de comportamiento que completan el catálogo GoF en LogiSmart. Cada patrón resolvió una necesidad concreta del dominio logístico:

  • Iterator permite recorrer tres tipos de colecciones heterogéneas (array, lista enlazada, mapa) con el mismo código cliente, desacoplando los algoritmos de procesamiento de las estructuras de datos que los contienen.
  • Mediator convierte el pipeline CentroDistribucionMediator→ValidadorEnvioMediator→Pago→Notificación→Auditoría de un grafo denso de referencias cruzadas a un hub de eventos: cualquier componente puede cambiarse sin tocar a los demás.
  • Memento provee auditoría completa del ciclo de vida de un envío con navegación bidireccional, resolviendo la necesidad de operadores de revisar cualquier estado histórico sin exponer los campos privados de Envio.
  • Observer convierte a Envio en un sujeto reactivo: cuatro sistemas (centro, notificación, auditoría, dashboard) reaccionan automáticamente a cada transición de estado sin que el envío los conozca directamente.

Los cuatro patrones se integran en SistemaLogisticaEventDriven, donde colaboran en cada ciclo de procesamiento: Memento guarda el estado antes y después, Iterator incorpora el envío a la colección, Observer suscribe los reactivos, y Mediator dispara el pipeline completo. Esta arquitectura event-driven permite agregar nuevos pasos al pipeline, nuevos tipos de colección o nuevos observadores sin modificar las clases existentes - cumpliendo el principio Open/Closed en sus cuatro dimensiones simultáneamente.

El análisis del trade-off arquitectónico (ubicación de ObservadorEnvio y MementoEnvio en dominio) documentó formalmente la tensión entre cohesión de patrón GoF y autonomía de la capa de dominio, optando por DIP y arquitectura limpia como prioridad.
