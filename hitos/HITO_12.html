Actividad 1 · State

State: máquina de estados del envío

Envio actúa como contexto. El campo estadoGoF delega comportamiento en objetos estado, evitando condicionales por texto para operaciones como validar, entregar, retener, devolver y reclamar.

public void validar()  { estadoGoF.validar(this); }

Actividad 2 · Strategy

Strategy: cálculo de costo intercambiable

El envío ahora puede recibir una EstrategiaCalculoCosto en runtime y ejecutar calcularCostoConEstrategia(). Esto separa la política de cálculo del objeto que contiene los datos logísticos.

La consigna usaba Math.random() * 500 para distancia. Se reemplazó por 500 km fijos para que los tests sean verificables y repetibles.
El campo tipo (URGENTE / EXPRESS / NORMAL) se construye vía EnvioBuilder.tipo(): los constructores especializados de Hito 10, 11 y 12 fueron consolidados en el Builder para eliminar el anti-patrón de constructores telescópicos.
Actividad 3 · Template Method

Template Method: proceso estándar de envío

ProcesoEnvio define el algoritmo final procesarEnvio(Envio) y delega los pasos variables a subclases. El orden queda cerrado: validar, calcular costo, procesar pago y notificar.

GoF · Template Method
public abstract class ProcesoEnvio {
    public final void procesarEnvio(Envio envio) {
        validar(envio);
        calcularCosto(envio);
        procesarPago(envio);
        notificar(envio);
    }

    protected abstract void validar(Envio envio);
    protected abstract void calcularCosto(Envio envio);
    protected abstract void procesarPago(Envio envio);
    protected abstract void notificar(Envio envio);
}

Actividad 5 · Integración

SistemaLogisticaAvanzada

La clase de integración ejecuta los cuatro patrones en un flujo único: State avanza el envío, Strategy calcula costo, Template Method procesa el envío y Visitor analiza la red de distribución.

public double procesarEnvio(Envio envio, EstrategiaCalculoCosto estrategia, ProcesoEnvio proceso) {
    envio.cambiarEstado(new EstadoConfirmado());
    envio.validar();
    envio.entregar();
    envio.establecerEstrategia(estrategia);
    double costo = envio.calcularCostoConEstrategia();
    proceso.procesarEnvio(envio);
    envio.entregar();
    return costo;
}

Análisis

Ventajas y desventajas por patrón

Cada patrón resuelve un problema concreto del dominio e introduce compromisos. Se analiza la decisión aplicada a LogiSmart, no solo la teoría general.

State

VentajasDesventajas
Elimina los condicionales por texto de estado (if estado.equals("EN_TRANSITO")): cada estado encapsula su propio comportamiento.Genera una clase por estado (9 estados = 9 clases), aumentando el número de archivos a mantener.
Las transiciones inválidas se manejan localmente en cada estado, haciendo la máquina explícita y verificable.La lógica de transición queda dispersa entre las clases de estado; ver el flujo completo exige recorrer varias clases.
Agregar un estado nuevo no rompe los existentes; Envio solo delega en estadoGoF.El contexto y los estados quedan acoplados: cada estado necesita conocer a qué estados puede transicionar.

Strategy

VentajasDesventajas
Permite intercambiar el algoritmo de cálculo de costo en runtime (establecerEstrategia()) sin tocar Envio.El cliente debe conocer las estrategias disponibles para elegir la adecuada.
Cada fórmula (distancia, peso, urgencia, volumen, híbrida) queda aislada y testeable por separado.Aumenta el número de objetos; estrategias muy simples pueden no justificar una clase propia.
La estrategia híbrida demuestra composición: combina estrategias existentes sin duplicar lógica.Si las estrategias necesitan muchos datos del contexto, la interfaz tiende a crecer y a filtrar detalles del Envio.

Template Method

VentajasDesventajas
Fija el orden del algoritmo en procesarEnvio() (validar, calcular, pagar, notificar) y evita que las subclases lo alteren al declararlo final.Usa herencia, no composición: una subclase está atada a la clase base y no puede cambiar de jerarquía en runtime.
Reutiliza el esqueleto común y solo obliga a implementar los pasos variables, reduciendo duplicación.Si los pasos variables crecen, la clase base puede volverse rígida y difícil de extender sin romper subclases.
El orden cerrado garantiza que ningún proceso (nacional, internacional, urgente) se saltee un paso obligatorio.Menos flexible que Strategy: el comportamiento se elige por subclase, no por inyección dinámica.

Visitor

VentajasDesventajas
Permite agregar operaciones nuevas sobre la red de distribución (ocupación, reporte, costo, puntos críticos) sin modificar las clases de nodo.Lo inverso es costoso: agregar un tipo de nodo nuevo obliga a tocar todos los visitors existentes.
Concentra cada operación en una sola clase visitor, en vez de dispersarla por todos los nodos.El doble dispatch (aceptar/visitar) es indirecto y menos intuitivo de leer al principio.
Separa el algoritmo de la estructura: la jerarquía de nodos solo sabe aceptar visitantes.El visitor suele necesitar acceso al estado interno de los nodos, lo que puede romper su encapsulamiento.

Conclusión

El Hito 12 cierra el catálogo GoF con cuatro patrones de comportamiento aplicados sobre puntos reales del dominio. La solución agrega comportamiento sin romper hitos previos, mantiene compatibilidad con Observer/Memento y deja tests deterministas para validar la integración.
