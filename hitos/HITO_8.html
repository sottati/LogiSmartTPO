Actividad 2 · Adapter

Implementación de Adapters

Se implementaron adapters para envíos y pagos. El cliente depende de interfaces internas, no de APIs externas.

Interfaz esperada por LogiSmart

public interface ProveedorEnvio {
    boolean crearEnvio(Envio envio);
    String obtenerEstado(String numeroSeguimiento);
    double calcularCosto(Envio envio);
    String obtenerNombre();
}

Adapters implementados

Rol dentro del patrón: ProveedorEnvio y ProveedorPago son las interfaces objetivo que entiende LogiSmart; DHLAPI, FedExAPI, UPSConnector, PayPalAPI y StripeAPI representan APIs externas incompatibles; cada Adapter* traduce nombres, tipos y unidades hacia el contrato interno.
public class AdapterDHL implements ProveedorEnvio {
    private final DHLAPI dhlAPI = new DHLAPI();

    public boolean crearEnvio(Envio envio) {
        String codigo = dhlAPI.registrarPaquete(
            envio.getOrigen(), envio.getDestino(), envio.getPeso());
        envio.getSeguimiento().actualizarEstado("DHL:" + codigo);
        return true;
    }

    public double calcularCosto(Envio envio) {
        return dhlAPI.calcularTarifa(
            envio.getOrigen(), envio.getDestino(), envio.getPeso());
    }
}
public class AdapterStripe implements ProveedorPago {
    private final StripeAPI stripeAPI = new StripeAPI();

    public boolean procesarPago(double monto, String referencia) {
        return stripeAPI.charge(monto * 100, referencia);
    }

    public String obtenerEstado(String idTransaccion) {
        String estado = stripeAPI.getChargeStatus(idTransaccion);
        return estado.equals("succeeded") ? "COMPLETADA" : "PENDIENTE";
    }
}

Actividad 4 · Bridge

Implementación de Bridge para reportes

La abstracción Reporte define el contenido. La implementación GeneradorReporte define el formato.

Rol dentro del patrón: Reporte es la abstracción; ReporteEnvios, ReporteIngresos y ReporteDesempenoProveedores son abstracciones refinadas; GeneradorReporte es el implementador; GeneradorPDF, GeneradorExcel, GeneradorJSON y GeneradorCSV son implementadores concretos. Así se evita crear una clase por cada combinación tipo/formato.
public interface GeneradorReporte {
    String formatear(String contenido);
    String obtenerExtension();
}

public class GeneradorJSON implements GeneradorReporte {
    public String formatear(String contenido) {
        return "{\"reporte\": \"" + contenido + "\"}";
    }
    public String obtenerExtension() { return "json"; }
}
public abstract class Reporte {
    protected GeneradorReporte generador;

    public String generar() {
        return generador.formatear(generarContenido());
    }

    public void setGenerador(GeneradorReporte generador) {
        this.generador = generador;
    }

    public abstract String generarContenido();
}

Actividad 6 · Composite

Implementación de Composite para centros

CentroDistribucion define la interfaz común. CentroRegional contiene hijos y PuntoEntrega actúa como hoja.

Rol dentro del patrón: CentroDistribucion es el componente común; CentroRegional es el composite porque contiene otros CentroDistribucion; PuntoEntrega es la hoja. Los niveles Nacional, Regional y Local se representan con instancias anidadas de CentroRegional, por eso no hace falta crear una clase por nivel.
public abstract class CentroDistribucion {
    protected String nombre;
    protected String ubicacion;
    protected String codigo;

    public abstract int obtenerCapacidad();
    public abstract int obtenerOcupacion();
    public abstract void mostrar(int profundidad);

    public double obtenerPorcentajeOcupacion() {
        return (double) obtenerOcupacion() / obtenerCapacidad() * 100;
    }
}
public class CentroRegional extends CentroDistribucion {
    private final List<CentroDistribucion> subcentros = new ArrayList<>();

    public int obtenerCapacidad() {
        int total = 0;
        for (CentroDistribucion centro : subcentros) {
            total += centro.obtenerCapacidad();
        }
        return total;
    }
}

Actividad 7 · Integración

ServicioLogisticaUnificado

El servicio integra Adapter, Bridge y Composite en un flujo único.

Integración específica: el cliente no instancia APIs externas ni reportes concretos. Llama a ServicioLogisticaUnificado; el servicio resuelve el adapter por nombre de proveedor, mantiene los envíos registrados, expone el centro raíz composite y arma el bridge reporte/formato cuando se solicita un reporte.
public class ServicioLogisticaUnificado {
    private final Map<String, ProveedorEnvio> proveedoresEnvio = new HashMap<>();
    private final Map<String, ProveedorPago> proveedoresPago = new HashMap<>();
    private final CentroDistribucion centroDistribucion;
    private final List<Envio> enviosRegistrados = new ArrayList<>();

    public boolean crearEnvio(String nombreProveedor, Envio envio) {
        ProveedorEnvio proveedor = proveedoresEnvio.get(nombreProveedor); // Adapter
        if (proveedor.crearEnvio(envio)) {
            enviosRegistrados.add(envio);
            return true;
        }
        return false;
    }

    public Reporte generarReporte(String tipoReporte, String formato) {
        GeneradorReporte generador = obtenerGenerador(formato); // Bridge
        return new ReporteEnvios(generador, enviosRegistrados);
    }

    public CentroDistribucion obtenerCentroDistribucion() { // Composite
        return centroDistribucion;
    }
}
