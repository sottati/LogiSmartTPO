Patron 1

Data Mapper

Desacopla el modelo de dominio del esquema de base de datos. Cada mapper es responsable de traducir entre el objeto del dominio y la fila de la tabla, sin que el objeto de dominio sepa nada de SQL.

Diseno

  • 4 interfaces en mapper/: EnvioMapper, ClienteMapper, CentroDistribucionMapper, CobroMapper.
  • 4 implementaciones SQL en mapper/sql/ con PreparedStatement — compilan con el JDK.
  • Cada mapper: insertar, actualizar, eliminar(String id), buscarPorId(String id).
  • EnvioMapperSQL: buscarPorId reconstruye el Envio via EnvioBuilder — fidelidad al dominio real.
  • Columna id es VARCHAR en todas las tablas (el dominio usa String id).

Decisiones de diseno

  • Reutilizacion: Envio, ClienteFinal y Cobro ya existian. Solo se crea CentroDistribucion (persistencia) nueva.
  • ClienteMapper: persiste subconjunto id, nombre, email, telefono. Ignora passwordHash, rol, estado y permisos (responsabilidad del modulo de autenticacion).
  • CobroMapper: incluye envioId (campo aditivo de Hito 13). El estado permanece String — no se introduce enum EstadoPago para no romper el comportamiento testeado.
  • SQL ilustrativo: compila, demuestra dominio de JDBC; los tests usan implementaciones en memoria.

Implementacion clave

// EnvioMapperSQL — reconstruccion via Builder
public Envio buscarPorId(String id) {
    String sql = "SELECT id, origen, destino, peso, estado, costo, metodo_pago, tipo "
               + "FROM envios WHERE id=?";
    try (PreparedStatement ps = connection.prepareStatement(sql)) {
        ps.setString(1, id);
        try (ResultSet rs = ps.executeQuery()) {
            if (rs.next()) {
                return new Envio.EnvioBuilder(
                        rs.getString("id"),
                        rs.getString("origen"),
                        rs.getString("destino"))
                    .peso(rs.getDouble("peso"))
                    .estado(rs.getString("estado"))
                    .costo(rs.getDouble("costo"))
                    .build();
            }
            return null;
        }
    }
}
Patron 2

Repository

Abstrae el acceso a colecciones de objetos del dominio. El cliente ve una coleccion tipada con metodos orientados al negocio; la implementacion puede ser en memoria, SQL o cualquier otra.

Jerarquia de interfaces

  • Repositorio<T>: generica — guardar, obtener(int), obtenerTodos, eliminar(int).
  • RepositorioEnvio: extiende con obtener(String) + buscarPorEstado(String).
  • RepositorioCliente: extiende con obtener(String) + buscarPorNombre(String).
  • RepositorioPago: extiende con obtener(String) + buscarPorEnvio(String).
  • RepositorioCentro: usa int id de la interfaz generica + buscarPorUbicacion.

Implementaciones

  • En memoria (repositorio/memoria/): HashMap interno, deterministas, testeadas.
  • SQL (repositorio/sql/): delegan en los mappers SQL, compilan, no se ejecutan en tests.
  • Divergencia documentada: obtener(int) retorna null para entidades con String id; el overload obtener(String) es el que se usa.

Patron 3

Unit of Work

Agrupa todos los cambios realizados durante una operacion de negocio y los confirma como una unica transaccion atomica. Evita inconsistencias parciales y reduce el numero de roundtrips a la base de datos.

Diseno

  • Tres listas: nuevos, modificados, eliminados.
  • registrarNuevo/Modificado/Eliminado(Object): encola el cambio sin ejecutarlo.
  • commit(): INSERT → UPDATE → DELETE (orden que preserva integridad referencial); luego vacia las listas.
  • rollback(): descarta todo sin ejecutar nada — semantica de abort.
  • Contadores publicos (cantidadNuevos(), etc.) para tests y observabilidad.

Integracion con la Facade

  • LogisticaFacade.procesarEnvioCompleto: registra Envio y Cobro como nuevos → commit() → persiste en repos en memoria.
  • Si el commit falla (excepcion en produccion), se puede llamar rollback() para limpiar el estado.
  • El UoW es stateful por diseno: se crea por operacion o se reutiliza limpiandolo.

Implementacion clave

public void commit() {
    System.out.println("[UnitOfWork] commit() iniciado");
    for (Object obj : nuevos)      System.out.println("  [INSERT] " + obj);
    for (Object obj : modificados) System.out.println("  [UPDATE] " + obj);
    for (Object obj : eliminados)  System.out.println("  [DELETE] " + obj);
    int total = nuevos.size() + modificados.size() + eliminados.size();
    limpiar();
    System.out.println("[UnitOfWork] commit() completado (" + total + " ops)");
}

public void rollback() {
    int descartados = cantidadTotal();
    limpiar();
    System.out.println("[UnitOfWork] rollback() — " + descartados + " cambios descartados.");
}
Patron 4

Lazy Load

Difiere la carga de datos costosos hasta el momento en que realmente se necesitan. Reduce la cantidad de objetos cargados en memoria y los accesos a la base de datos cuando el dato puede no usarse.

Tres proxies implementados

  • ClienteLazyProxy: carga ClienteFinal desde RepositorioClienteMemoria al primer getCliente().
  • CentroDistribucionLazyProxy: carga la entidad de persistencia CentroDistribucion al primer getCentro().
  • HistorialEnviosLazyProxy: carga y filtra la lista de envios del cliente al primer getHistorial(). Filtra por clienteId en el id del envio.

Garantias del patron

  • No carga al crear: el proxy solo guarda el id y la referencia al repositorio.
  • Carga al primer acceso: llamada al repositorio solo cuando se invoca el getter.
  • No recarga: segunda llamada retorna la instancia ya cargada (cache en campo privado).
  • estaCargado(): observable publico para tests y diagnostico.

Diagrama de secuencia

sequenceDiagram
    participant App as Aplicacion
    participant P as ClienteLazyProxy
    participant R as RepositorioClienteMemoria

    App->>P: new ClienteLazyProxy(repo, "CLI-001")

    P-->>App: ClienteFinal [sin ir al repo]

Implementacion clave

public class ClienteLazyProxy {
    private final RepositorioClienteMemoria repositorio;
    private final String clienteId;
    private ClienteFinal clienteCargado = null;   // null = no cargado

    public ClienteFinal getCliente() {
        if (clienteCargado == null) {
            clienteCargado = repositorio.obtener(clienteId);  // carga lazy
        }
        return clienteCargado;
    }

    public boolean estaCargado() { return clienteCargado != null; }
}

SistemaPersistencia y LogisticaFacade

La LogisticaFacade coordina los cuatro servicios de aplicacion y el UnitOfWork. SistemaPersistencia es la demo de integracion que muestra la arquitectura completa funcionando.

CentroAssembler — mitigacion de desfasaje

El CentroDistribucion del dominio es un Composite abstracto cuyos atributos capacidad y ocupacion se calculan sumando los hijos. No puede ser una fila de tabla. Solucion: entidad de persistencia plana + CentroAssembler que la genera como proyeccion del Composite al momento de guardar. El Composite es la unica fuente de verdad — el snapshot nunca puede desincronizarse porque se recalcula cada vez. 
// CentroAssembler — proyeccion Composite -> entidad plana
public static CentroDistribucion aPersistencia(
        String id,
        composite.centro.CentroDistribucion c) {
    return new CentroDistribucion(
            id, c.getNombre(), c.getUbicacion(), c.getCodigo(),
            c.obtenerCapacidad(),   // calculado por el Composite
            c.obtenerOcupacion()    // calculado por el Composite
    );
}

procesarEnvioCompleto — flujo integrado

public void procesarEnvioCompleto(Envio envio, Cobro pago) {
    pago.setEnvioId(envio.getId());        // vincula pago con envio
    unitOfWork.registrarNuevo(envio);      // encola INSERT envio
    unitOfWork.registrarNuevo(pago);       // encola INSERT pago
    unitOfWork.commit();                   // simula transaccion atomica
    servicioEnvios.crearEnvio(envio);      // persiste en repo memoria
    servicioPagos.crearPago(pago);         // persiste en repo memoria
}

Arquitectura de la capa de persistencia

Cinco capas verticales: Presentacion → Aplicacion → Dominio → Persistencia → Datos. El dominio no depende de la persistencia; la persistencia depende del dominio (regla de dependencia).

Analisis

Ventajas y Desventajas

Ventajas

PatronVentaja principal
Data MapperEl dominio no sabe nada de SQL; se puede cambiar el esquema sin tocar las clases de negocio.
RepositoryEl cliente ve una coleccion tipada con metodos del negocio; la implementacion es intercambiable (memoria vs SQL).
Unit of WorkAtomicidad: o se guardan todos los cambios o ninguno. Reduce roundtrips a la BD.
Lazy LoadSolo carga los datos que se usan efectivamente; reduce memoria y latencia en grafos profundos.

Desventajas y trade-offs

PatronTrade-off
Data MapperMayor cantidad de clases; el mapeo puede volverse verbose en entidades con muchos campos.
RepositoryTension entre interfaz generica (int id) y dominio real (String id) — requiere overloads documentados.
Unit of WorkEstado mutable compartido; si no se limpia entre operaciones puede acumular cambios no deseados.
Lazy LoadEl primer acceso tiene latencia; en loops puede causar el problema N+1 si no se gestiona con cuidado.
Desfasaje Composite ↔ persistencia: el CentroDistribucion del dominio es abstracto y jerarquico; su modelo de persistencia es plano. El riesgo de divergencia se mitiga haciendo que la entidad plana sea generada como proyeccion (snapshot) por CentroAssembler al momento de guardar — nunca se edita directamente. Por construccion, el snapshot siempre refleja el estado actual del Composite.
Cierre del TPO

Conclusion

El Hito 13 cierra el ciclo de 13 hitos del TPO LogiSmart aplicando los cuatro patrones de acceso a datos sobre un dominio ya construido. La decision central fue reutilizar las entidades existentes en lugar de crear nuevas — lo que convierte el hito en un ejercicio de integracion real, no de ejemplo aislado.

  • Reutilizacion como decision de diseno superior: la consigna traia ejemplos con entidades nuevas; el proyecto ya tenia casi todas. Adaptarlas (en lugar de duplicarlas) es mas realista y demuestra mayor dominio del patron.
  • Una pregunta del Hito 2 que se cierra aqui: "?Pago clase o atributo?" — resuelta con Cobro, confirmada al mapearlo en el Hito 13.
  • Mas alla de lo pedido: la consigna pedia diseno en papel; correr el codigo de verdad (44 tests en verde, regresion 0) genero aprendizaje adicional y evidencia ejecutable del diseno.
  • Disciplina de 13 hitos: "solo agregar, nunca romper" — los 148 tests de hitos anteriores siguen en verde.
