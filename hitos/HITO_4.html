Hallazgos

Al revisar el modelo, la mayoria de las clases del dominio tienen alta cohesion: sus metodos estan relacionados con una sola responsabilidad de negocio. Sin embargo, se identificaron cuatro puntos concretos de mejora:

  • BCrypt en el paquete de dominio: es infraestructura tecnica y no deberia estar aqui. Su presencia acopla el dominio a la decision de usar hashing, lo que dificulta cambiar la implementacion de hashing.
  • Reporte.exportar() / compartir(): son operaciones de I/O ajenas al dominio. Deben moverse a un ReporteService en la capa de aplicacion.
  • SeguimientoEnvio.publicarTracking(): notificar a sistemas externos no es una responsabilidad del dominio. Se extrae a TrackingService, igual que NotificadorDeEnvios en el ejemplo de clase.
  • ETA.recalcular(): le asigna a ETA la responsabilidad de recalcularse, pero ETA no tiene la informacion necesaria para hacerlo (necesita datos del recorrido que pertenecen a SeguimientoEnvio). Segun Expert, esa responsabilidad es de SeguimientoEnvio.

2.1 Calcular la distancia total de una ruta

Experto identificado: Ruta Ruta contiene la lista de PuntoEntrega en orden, cada uno con lat y lng. Es el unico objeto con acceso directo a todos los segmentos del recorrido. Delega el acceso a coordenadas individuales a PuntoEntrega.
PreguntaRespuesta
Que informacion necesitaLista ordenada de paradas con lat/lng de cada punto

Cambio propuesto: agregar calcularDistanciaTotal(): double a Ruta, calculando la distancia entre cada par de paradas consecutivas usando sus coordenadas.

Aplicacion del Patron Creator

Identificacion de quien debe crear cada clase aplicando las reglas de Creator: Contencion (B compone A), Agregacion (B agrega A), Datos (B tiene los datos para inicializar A), Uso Frecuente (B usa A frecuentemente).

3.2 Problemas identificados

Problema 1: Envio no crea SeguimientoEnvio ni Entrega internamente En la implementacion actual, el constructor de Envio recibe SeguimientoEnvio y Entrega como parametros externos. Segun Creator (regla de Contencion), Envio deberia crearlos internamente porque los compone por ciclo de vida. Recibirlos desde afuera obliga al creador externo (OperadorLogistico) a conocer los detalles de construccion de objetos internos del Envio, aumentando el acoplamiento innecesariamente.
Problema 2: PuntoEntrega se construye fuera de Ruta Ruta.agregarParada(PuntoEntrega punto) recibe el objeto ya construido desde afuera. Segun Creator, Ruta deberia ofrecer un metodo factory que cree el PuntoEntrega internamente a partir de los datos necesarios (direccion, coordenadas, ventana horaria).

3.3 Pseudocodigo de creacion - soluciones propuestas

Caso 1: Envio crea SeguimientoEnvio y Entrega internamente (Creator - Contencion)

// PROPUESTA: Envio crea sus partes internas
public class Envio {
    private SeguimientoEnvio seguimiento;
    private Entrega entrega;

    public Envio(String id, String empresaId, String prioridad, LocalDateTime fechaProgramada) {
        this.id = id;
        this.empresaId = empresaId;
        this.estado = "PENDIENTE";
        this.prioridad = prioridad;
        this.fechaProgramada = fechaProgramada;
        this.ordenes = new ArrayList<>();
        // Envio es Creator de sus componentes internos por composicion
        this.seguimiento = new SeguimientoEnvio(id + "-seg", "PENDIENTE");
        this.entrega     = new Entrega(id + "-ent");
    }
}

Caso 2: Ruta encapsula la creacion de PuntoEntrega (Creator - Contencion)

// PROPUESTA: Ruta ofrece factory method para agregar paradas
public class Ruta {
    // Sobrecarga: acepta datos primitivos y crea PuntoEntrega internamente
    public void agregarParada(String direccion, double lat, double lng,
                              String ventanaHorario, int ordenParada) {
        PuntoEntrega punto = new PuntoEntrega(direccion, lat, lng, ventanaHorario, ordenParada);
        this.paradas.add(punto);
    }
}

Caso 3: SeguimientoEnvio crea PosicionGPS (Creator - Contencion)

// SeguimientoEnvio registra nueva posicion, creandola internamente (Creator - Contencion)
public class SeguimientoEnvio {
    public void registrarPosicion(double lat, double lng, double precision) {
        // SeguimientoEnvio es Creator de PosicionGPS (lo contiene)
        PosicionGPS pos = new PosicionGPS(lat, lng, LocalDateTime.now(), precision);
        this.historialPosiciones.add(pos);
        this.ultimaActualizacion = LocalDateTime.now();
    }
}

Caso 4: OperadorLogistico crea Envio y Ruta (Creator - Datos)

// OperadorLogistico tiene todos los datos para crear Envio y Ruta
public class OperadorLogistico extends Usuario {

    public Envio crearEnvio(String prioridad, LocalDateTime fecha) {
        // Operador es Creator de Envio: tiene empresaId y datos de negocio
        return new Envio(UUID.randomUUID().toString(), this.empresaId, prioridad, fecha);
    }

    public Ruta planificarRuta(Vehiculo vehiculo, Transportista transportista) {
        // Operador es Creator de Ruta: tiene vehiculo y transportista disponibles
        Ruta ruta = new Ruta(UUID.randomUUID().toString(), 0.0, 0, "PLANIFICADA");
        ruta.asignarVehiculo(vehiculo);
        ruta.asignarTransportista(transportista);
        return ruta;
    }
}

4.4 Reduccion de acoplamiento - Low Coupling aplicado

El modelo ya aplica correctamente la inyeccion de dependencias en los casos criticos: Ruta.asignarVehiculo(Vehiculo v) y Ruta.asignarTransportista(Transportista t) reciben las dependencias como parametros en lugar de instanciarlas internamente. Esto mantiene a Ruta desacoplada del ciclo de vida de Vehiculo y Transportista.

El uso de empresaId: String en Envio (en lugar de una referencia directa a Empresa) es una decision deliberada de bajo acoplamiento: Envio no necesita toda la informacion de Empresa, solo su identificador para trazabilidad.

Propuesta adicional - mover BCrypt fuera del dominio: Mover BCrypt.java al paquete com.logismart.util. Asi el paquete de dominio queda con una unica responsabilidad bien definida (logica de negocio) y no depende de una clase tecnica de hashing.
