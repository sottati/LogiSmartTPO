Contexto e integración

El Hito 10 introduce tres patrones de comportamiento del catálogo GOF. Los patrones de comportamiento modelan cómo los objetos interactúan y distribuyen responsabilidades, complementando los patrones creacionales (Hitos 6-7) y estructurales (Hitos 8-9) ya implementados.

Chain of Responsibility

Cadena de 5 validadores que aprueba o rechaza un envío sin que el cliente conozca cuál eslabón actúa.

Command

Encapsula operaciones como objetos con soporte de undo/redo mediante un historial con puntero.

Interpreter

Motor de reglas de negocio con expresiones booleanas terminales (Origen, Destino, Peso, Costo) y operadores AND, OR, NOT.

Actividad 1 · Chain of Responsibility

Identificación de candidatos

Se analizaron los flujos de validación del sistema para identificar dónde múltiples reglas independientes deben aplicarse en secuencia sobre un mismo objeto.

GoF · Chain of Responsibility

Candidatos identificados

CandidatoProblema actualPor qué Chain resuelve
Validación de envíosLas reglas de datos, inventario, pago, seguridad y capacidad estaban mezcladas en un solo método o clase.Cada regla pasa a un eslabón independiente; agregar, quitar o reordenar reglas no requiere tocar las demás.
Aprobación de créditoRequiere pasar por sucursal, regional y central en orden.La solicitud sube la cadena hasta que algún nivel la aprueba o rechaza.
Filtros de logMensajes de distinta severidad deben procesarse por manejadores distintos.Cada handler decide si procesa o delega según el nivel.

Diseño de la cadena de validación

Se eligió el candidato "Validación de envíos" por ser el más representativo del dominio. La cadena se diseñó con 5 validadores en orden de costo creciente: los más baratos (datos, inventario) actúan primero para hacer fallo rápido.

PosiciónValidadorCondición de rechazoJustificación de orden
1ValidadorDatosOrigen o destino vacío, o peso ≤ 0Datos básicos: costo O(1), falla más frecuente.
2ValidadorInventarioProductoId sin stock en SistemaInventarioConsulta de mapa en memoria: O(1).
3ValidadorPagoCosto ≤ 0 o método de pago vacíoRegla de negocio pura, sin I/O.
4ValidadorSeguridadDestino contiene "Restringido"Requiere política de zonas; va después de datos básicos.
5ValidadorCapacidadPeso supera capacidadDisponible en SistemaCapacidadEstado global del sistema; se evalúa al final.
Decisión de diseño: La clase CadenaValidadores actúa como Facade de la cadena: recibe las dependencias externas (SistemaInventario, SistemaCapacidad) y construye el orden de los eslabones en su constructor. El cliente solo llama validarEnvio(Envio) y desconoce la estructura interna.
Actividad 2 · Chain of Responsibility

Implementación

Se implementaron la clase abstracta ValidadorEnvio, los cinco validadores concretos y la clase de ensamblado CadenaValidadores. Los sistemas de soporte (SistemaInventario, SistemaCapacidad) se ubicaron en el mismo paquete para evitar dependencias ascendentes.

GoF · Chain of Responsibility
// Handler abstracto
public abstract class ValidadorEnvio {
    protected ValidadorEnvio siguiente;

    public void setSiguiente(ValidadorEnvio sig) {
        this.siguiente = sig;
    }

    public abstract boolean validar(Envio envio);
    public abstract String obtenerNombre();
}
// Concrete handler
public class ValidadorDatos extends ValidadorEnvio {
    public boolean validar(Envio envio) {
        if (envio.getOrigen() == null
                || envio.getOrigen().isEmpty()) return false;
        if (envio.getDestino() == null
                || envio.getDestino().isEmpty()) return false;
        if (envio.getPeso() <= 0) return false;

        System.out.println("  ✓ Datos válidos");
        return siguiente == null || siguiente.validar(envio);
    }
    public String obtenerNombre() { return "ValidadorDatos"; }
}
// Ensamblado de la cadena
public class CadenaValidadores {
    private final ValidadorEnvio primerValidador;

    public CadenaValidadores(SistemaInventario inv, SistemaCapacidad cap) {
        ValidadorEnvio v1 = new ValidadorDatos();
        ValidadorEnvio v2 = new ValidadorInventario(inv);
        ValidadorEnvio v3 = new ValidadorPago();
        ValidadorEnvio v4 = new ValidadorSeguridad();
        ValidadorEnvio v5 = new ValidadorCapacidad(cap);

        v1.setSiguiente(v2); v2.setSiguiente(v3);
        v3.setSiguiente(v4); v4.setSiguiente(v5);
        this.primerValidador = v1;
    }

    public boolean validarEnvio(Envio envio) {
        return primerValidador.validar(envio);
    }
}

Patrón de delegación

Cada validador aprueba su regla y delega con return siguiente == null || siguiente.validar(envio). Si no hay siguiente, retorna true (fin de cadena = envío aprobado). Si rechaza, corta la cadena y retorna false inmediatamente sin evaluar los eslabones restantes.

Actividad 3 · Command

Identificación de operaciones reversibles

Se relevaron las operaciones sobre envíos que: (a) deben registrarse con trazabilidad, (b) pueden necesitar ser revertidas por el operador, o (c) pueden encolar para procesamiento diferido.

Diseño del historial undo/redo

Se eligió un historial lineal con índice entero (indiceActual) en lugar de dos pilas separadas. Este enfoque permite:

  • Undo: decrementar el índice y llamar deshacer() en el comando señalado.
  • Redo: incrementar el índice y volver a llamar ejecutar().
  • Nuevo comando: podar los comandos posteriores al índice actual antes de agregar, manteniendo un árbol lineal.
  • Auditoría: el historial completo es visible en todo momento con mostrarHistorial().
Actividad 4 · Command

Implementación

Se implementaron la interfaz Comando, el receiver ServicioEnvios, el invoker ColaComandos y cinco comandos concretos, cada uno capturando el estado previo necesario para revertir su efecto.

GoF · Command
// Interface
public interface Comando {
    void   ejecutar();
    void   deshacer();
    String obtenerDescripcion();
}
// Concrete Command — guarda estado previo
public class ComandoCambiarMetodoPago implements Comando {
    private final ServicioEnvios servicio;
    private final String numero;
    private final String nuevoMetodo;
    private String metodoAnterior;

    public void ejecutar() {
        metodoAnterior = servicio.obtenerMetodoPago(numero);
        servicio.cambiarMetodoPago(numero, nuevoMetodo);
    }
    public void deshacer() {
        servicio.cambiarMetodoPago(numero, metodoAnterior);
    }
    public String obtenerDescripcion() {
        return "Cambiar pago " + numero + " → " + nuevoMetodo;
    }
}
// Invoker con undo/redo lineal
public class ColaComandos {
    private final List<Comando> historial = new ArrayList<>();
    private int indiceActual = -1;

    public void ejecutar(Comando c) {
        while (indiceActual < historial.size() - 1)   // podar futuro
            historial.remove(historial.size() - 1);
        c.ejecutar();
        historial.add(c);
        indiceActual++;
    }

    public void deshacer() {
        if (indiceActual >= 0)
            historial.get(indiceActual--).deshacer();
    }

    public void rehacer() {
        if (indiceActual < historial.size() - 1)
            historial.get(++indiceActual).ejecutar();
    }

    public int obtenerTamano()   { return historial.size(); }
    public int getIndiceActual() { return indiceActual; }
}

Actividad 5 · Interpreter

Definición de la gramática

Se diseñó un motor de reglas de negocio sobre envíos. Una "regla" es cualquier combinación booleana de condiciones atómicas sobre atributos del envío, representada como un árbol de expresiones.

GoF · Interpreter

Gramática BNF

Expresion   ::= Terminal | NoTerminal
Terminal    ::= ExpresionOrigen  "(" <string> ")"
              | ExpresionDestino "(" <string> ")"
              | ExpresionPeso    "(" <numero> "," <op> ")"
              | ExpresionCosto   "(" <numero> "," <op> ")"
              | ExpresionRestringido
NoTerminal  ::= ExpresionAND "(" Expresion "," Expresion ")"
              | ExpresionOR  "(" Expresion "," Expresion ")"
              | ExpresionNOT "(" Expresion ")"
op          ::= "<" | ">" | "="

Árbol de una regla compleja

La regla "ORIGEN = Buenos Aires AND COSTO > 100 AND NOT RESTRINGIDO" se construye como:

                     AND
                    /   \
                  AND    NOT
                 /   \     \
          ORIGEN   COSTO    RESTRINGIDO
         ="BA"     > 100

Actividad 6 · Interpreter

Implementación

Se implementaron la interfaz Expresion, cinco expresiones terminales y tres operadores no-terminales. Las expresiones terminales con operador (ExpresionPeso, ExpresionCosto) aceptan los operadores "<", ">" y "=".

GoF · Interpreter
// Interface — Context es el Envio
public interface Expresion {
    boolean evaluar(Envio envio);
}

// Terminal simple
public class ExpresionOrigen implements Expresion {
    private final String valor;
    public boolean evaluar(Envio e) {
        return valor.equals(e.getOrigen());
    }
}

// Terminal con operador
public class ExpresionPeso implements Expresion {
    private final double valor;
    private final String operador;
    public boolean evaluar(Envio e) {
        switch (operador) {
            case "<": return e.getPeso() <  valor;
            case ">": return e.getPeso() >  valor;
            case "=": return e.getPeso() == valor;
            default:  return false;
        }
    }
}
// No-terminales
public class ExpresionAND implements Expresion {
    private final Expresion izquierda, derecha;
    public boolean evaluar(Envio e) {
        return izquierda.evaluar(e) && derecha.evaluar(e);
    }
}

public class ExpresionOR implements Expresion {
    private final Expresion izquierda, derecha;
    public boolean evaluar(Envio e) {
        return izquierda.evaluar(e) || derecha.evaluar(e);
    }
}

public class ExpresionNOT implements Expresion {
    private final Expresion expresion;
    public boolean evaluar(Envio e) {
        return !expresion.evaluar(e);
    }
}

// Terminal especial
public class ExpresionRestringido implements Expresion {
    public boolean evaluar(Envio e) {
        return e.getDestino().contains("Restringido");
    }
}

Actividad 7 · Integración

Sistema logístico completo

SistemaLogisticaCompleto integra los tres patrones en un único flujo de procesamiento: la cadena valida el envío, el comando lo crea con historial reversible y el intérprete evalúa las reglas de negocio configuradas.

1. Chain

Valida el envío.
Si falla → retorna null.

2. Command

Crea el envío.
Guarda en historial.

3. Interpreter

Evalúa reglas.
Muestra resultado.

public class SistemaLogisticaCompleto {
    private final CadenaValidadores validadores;
    private final ColaComandos cola;
    private final ServicioEnvios servicio;
    private final Map<String, Expresion> reglas;

    public SistemaLogisticaCompleto() {
        SistemaInventario inventario = new SistemaInventario();
        SistemaCapacidad  capacidad  = new SistemaCapacidad(1000.0);
        this.validadores = new CadenaValidadores(inventario, capacidad);
        this.servicio    = new ServicioEnvios();
        this.cola        = new ColaComandos();
        this.reglas      = new HashMap<>();
        inicializarReglas();
    }

    public String procesarEnvio(Envio envio) {
        // 1. Chain of Responsibility: validar
        if (!validadores.validarEnvio(envio)) return null;

        // 2. Command: crear con historial reversible
        ComandoCrearEnvio cmd = new ComandoCrearEnvio(servicio, envio);
        cola.ejecutar(cmd);
        String numero = cmd.getNumeroSeguimiento();

        // 3. Interpreter: evaluar reglas de negocio
        reglas.forEach((k, v) ->
            System.out.println((v.evaluar(envio) ? "✓ " : "  ") + k));

        return numero;
    }

    public ServicioEnvios    getServicio()    { return servicio; }
    public ColaComandos      getCola()        { return cola; }
    public CadenaValidadores getValidadores() { return validadores; }
}

Conclusión

El Hito 10 incorpora tres patrones de comportamiento que mejoran la flexibilidad del sistema en dimensiones complementarias:

  • Chain of Responsibility desacopla el emisor de la validación de sus reglas concretas, permitiendo agregar o reordenar validadores sin modificar el cliente ni los eslabones existentes.
  • Command convierte cada operación en un objeto con historial auditable y soporte de undo/redo, aportando trazabilidad y la posibilidad de revertir cualquier acción del operador.
  • Interpreter externaliza las reglas de negocio en un motor de expresiones componibles, permitiendo definir nuevas reglas combinando terminales existentes sin modificar código.

Los tres patrones se integran en SistemaLogisticaCompleto, donde la cadena actúa como guardián de entrada, el comando registra la creación, y el intérprete clasifica el envío según políticas de negocio. El sistema queda preparado para incorporar los patrones de comportamiento del Hito 11 (Iterator, Observer, State, Strategy, Template Method) sin afectar lo implementado.
