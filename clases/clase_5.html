Asignación de Responsabilidades: Controller, Polymorphism, Pure Fabrication, Indirection, Protected Variations

Introducción

En la clase anterior vimos GRASP Parte I: Expert, Creator, Low Coupling y High Cohesion. Hoy veremos los 5 patrones restantes que nos permitirán diseñar sistemas más flexibles, mantenibles y escalables.

¿Por Qué Estos Patrones?

El software cambia. Los requisitos cambian. Las tecnologías cambian. Un buen diseño debe anticipar estos cambios y hacerlos fáciles de implementar sin romper el código existente.

Estos 5 patrones nos enseñan exactamente cómo hacer eso:

  • Controller: Quién maneja las solicitudes del usuario
  • Polymorphism: Usar interfaces para eliminar condicionales
  • Pure Fabrication: Inventar clases para salvar el diseño
  • Indirection: Desacoplar componentes a través de intermediarios
  • Protected Variations: Proteger el código de cambios futuros

1. Controller Pattern

El Problema

Cuando la interfaz de usuario (UI) maneja directamente la lógica de negocio, se crea un acoplamiento fuerte. Si cambias la UI, rompes la lógica. Si cambias la lógica, rompes la UI.

❌ INCORRECTO
class BibliotecaUI { void prestarLibro() { Libro libro = biblioteca.buscarLibro(isbn); if (libro.isDisponible()) { libro.setDisponible(false); miembro.agregarLibro(libro); registro.crear(miembro, libro); // ... más lógica } } }
✅ CORRECTO
class BibliotecaController { private Biblioteca biblioteca; public boolean prestarLibro( String isbn, int idMiembro) { Libro libro = biblioteca.buscarLibro(isbn); Miembro miembro = biblioteca.obtenerMiembro( idMiembro); return biblioteca.prestarLibro( libro, miembro); } }

La Solución

Crear una clase Controller que actúe como intermediaria entre la UI y el dominio. El Controller recibe las solicitudes, coordina las operaciones, y devuelve resultados.

Ventajas

  • Desacoplamiento: La UI no conoce los detalles del dominio
  • Testabilidad: Podemos testear el Controller sin la UI
  • Reutilización: El mismo Controller puede usarse con diferentes UIs
  • Mantenibilidad: Cambios en la UI no afectan el dominio

Tipos de Controllers

Tipo Descripción Ejemplo
Facade Controller Coordina múltiples objetos del dominio BibliotecaController
Use Case Controller Maneja un caso de uso específico PrestarLibroController
El Controller es como un "gerente" que coordina el trabajo

2. Polymorphism Pattern

El Problema

La proliferación de condicionales anidados hace el código frágil y difícil de mantener.

// ❌ INCORRECTO: Condicionales anidados if (usuario instanceof Cliente) { // Hacer algo para Cliente } else if (usuario instanceof Operador) { // Hacer algo para Operador } else if (usuario instanceof Administrador) { // Hacer algo para Administrador }

Problemas:

  • Violación del Principio Open/Closed
  • Cada nuevo tipo requiere modificar el código existente
  • Difícil de testear: necesitas testear cada rama

La Solución

Usar Polymorphism (polimorfismo). Cada tipo de usuario implementa su propio comportamiento. El código que usa los usuarios no necesita saber qué tipo son.

// ✅ CORRECTO: Polymorphism interface Usuario { boolean puedePrestarLibro(); boolean puedeModificarCatalogo(); } class Cliente implements Usuario { public boolean puedePrestarLibro() { return true; } public boolean puedeModificarCatalogo() { return false; } } class BibliotecaController { void prestarLibro(Usuario usuario, Libro libro) { if (usuario.puedePrestarLibro()) { // Realizar préstamo } } }

Ventajas

  • Extensibilidad: Agregar un nuevo tipo es fácil
  • Mantenibilidad: El código que usa usuarios no cambia
  • Testabilidad: Puedes crear usuarios mock
  • Claridad: El código expresa la intención claramente
Polymorphism es uno de los pilares de la POO

3. Pure Fabrication Pattern

El Problema

A veces, aplicar Expert y Creator nos lleva a clases con demasiadas responsabilidades. Es como si tuviéramos que forzar una responsabilidad en una clase que no le corresponde naturalmente.

Pregunta: ¿Quién debería ser responsable de enviar emails? ¿El Miembro? ¿La Biblioteca? Ninguno de los dos se siente "experto" en enviar emails.

La Solución

Pure Fabrication es inventar una clase que no existe en el dominio real, pero que es útil en el diseño. Es como contratar a un especialista externo para hacer un trabajo que nadie en la empresa quiere hacer.

❌ INCORRECTO
class Miembro { public void notificar(String msg) { // Conectar a SMTP // Autenticarse // Enviar email // Manejar errores // ... 50 líneas } }
✅ CORRECTO
class NotificadorDePrestamos { private ServicioDeEmail servicio; public void notificar( Miembro miembro, Libro libro) { String msg = "Préstamo: " + libro.getTitulo(); servicio.enviar( miembro.getEmail(), msg); } }

Ventajas

  • Separación de Responsabilidades: Cada clase tiene una responsabilidad clara
  • Reutilización: El Notificador puede usarse en otros contextos
  • Testabilidad: Podemos testear el Notificador independientemente
  • Mantenibilidad: Cambios en notificación no afectan Miembro

Ejemplos Comunes

Pure Fabrication es común en software profesional:

  • Logger - Registra eventos
  • EmailService - Envía emails
  • PaymentProcessor - Procesa pagos
  • ReportGenerator - Genera reportes
Pure Fabrication es una clase artificial pero esencial en el diseño

4. Indirection Pattern

El Problema

Acoplamiento fuerte entre componentes. Si la Biblioteca depende directamente de un ServicioDeEmail específico, cambiar el proveedor requiere modificar la Biblioteca.

// ❌ INCORRECTO: Acoplamiento fuerte class Biblioteca { private ServicioDeEmailGmail servicio; public void prestarLibro(Miembro m, Libro l) { servicio.enviarEmail(m.getEmail(), "Préstamo confirmado"); } } // Si cambias de Gmail a SendGrid, // tienes que cambiar Biblioteca

La Solución

Indirection (indirección): crear una interfaz que desacople los componentes. En lugar de que Biblioteca dependa de ServicioDeEmailGmail, depende de una interfaz Notificador.

// ✅ CORRECTO: Indirection con interfaz interface Notificador { void notificar(String email, String msg); } class Biblioteca { private Notificador notificador; public void prestarLibro(Miembro m, Libro l) { notificador.notificar(m.getEmail(), "Préstamo confirmado"); } } // Ahora puedes cambiar la implementación // sin tocar Biblioteca

Ventajas

  • Flexibilidad: Cambiar implementaciones es fácil
  • Testabilidad: Puedes usar mocks en tests
  • Desacoplamiento: Las clases no dependen de detalles
  • Escalabilidad: Agregar nuevas implementaciones es simple

Indirection con Inyección de Dependencias

// Inyección de dependencias en constructor class Biblioteca { private Notificador notificador; public Biblioteca(Notificador notificador) { this.notificador = notificador; } } // En la aplicación: Notificador notificador = new NotificadorEmail(); Biblioteca biblioteca = new Biblioteca(notificador); // Si quieres cambiar a SMS: Notificador notificador = new NotificadorSMS(); Biblioteca biblioteca = new Biblioteca(notificador);
Indirection es la base de la inyección de dependencias

5. Protected Variations Pattern

El Problema

¿Cómo protegemos nuestro código de cambios futuros que no podemos predecir? Sabemos que el software cambia. ¿Cómo diseñamos para anticipar estos cambios?

La Solución

Protected Variations (variaciones protegidas): identificar los puntos de variación (lugares donde es probable que haya cambios) y crear abstracciones que protejan el código.

Es como construir una casa con paredes modulares. Si necesitas cambiar la distribución, cambias las paredes, no los cimientos.

Ejemplo: Cálculo de Multas

Punto de variación: Cómo se calcula la multa. Podría cambiar: multa fija, multa por día, multa con descuentos, etc.

❌ INCORRECTO
class RegistroDePrestamo { public double calcularMulta( Date fechaDevolucion) { long dias = calcularDias( fechaDevolucion); return dias * 5.0; // Fija } } // Si cambias la política, // tienes que cambiar esta clase
✅ CORRECTO
interface CalculadorDeMulta { double calcular(long diasAtraso); } class RegistroDePrestamo { private CalculadorDeMulta calc; public double calcularMulta( Date fechaDevolucion) { long dias = calcularDias( fechaDevolucion); return calc.calcular(dias); } }

Ventajas

  • Anticipación: Identificas puntos de variación antes de cambios
  • Estabilidad: El código que usa abstracciones es estable
  • Flexibilidad: Cambios futuros son fáciles de implementar
  • Mantenibilidad: Menos código para cambiar

Puntos de Variación Comunes

Área Punto de Variación Solución
Algoritmos Cálculos, validaciones Strategy Pattern
Fuentes de datos BD, APIs, archivos Repository Pattern
Servicios externos Email, SMS, pagos Adapter Pattern
Políticas de negocio Multas, descuentos, permisos Strategy Pattern
Protected Variations es sobre ser proactivo, no reactivo

Resumen de los 5 Patrones

Patrón Problema Solución Beneficio
Controller ¿Quién maneja solicitudes? Clase intermediaria Desacoplamiento UI-Dominio
Polymorphism Condicionales anidados Interfaces e implementaciones Código flexible y extensible
Pure Fabrication Responsabilidades forzadas Clases artificiales útiles Separación de responsabilidades
Indirection Acoplamiento fuerte Interfaces que desacoplan Flexibilidad para cambios
Protected Variations Cambios futuros impredecibles Abstracciones en puntos de variación Código robusto y anticipatorio

Cómo Trabajan Juntos

Estos 5 patrones no son independientes. Trabajan juntos para crear un diseño flexible, mantenible y escalable:

Ejemplo Integrado: Un Controller usa Polymorphism para manejar diferentes tipos de usuarios. Delega responsabilidades a Pure Fabrication (NotificadorDePrestamos). Usa Indirection (interfaces) para desacoplar componentes. Y anticipa cambios futuros con Protected Variations (estrategias de cálculo de multas).
GRASP Parte I + GRASP Parte II = Diseño Robusto
