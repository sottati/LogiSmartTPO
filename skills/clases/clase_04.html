¿Qué es GRASP?

GRASP es un acrónimo que significa General Responsibility Assignment Software Patterns (Patrones Generales de Asignación de Responsabilidades en Software).

GRASP proporciona patrones probados para responder a la pregunta más importante en el diseño orientado a objetos:

¿A qué clase le asigno esta responsabilidad?

Por Qué GRASP Importa

La diferencia entre un buen diseño y uno malo no está en la tecnología que usamos, sino en cómo asignamos responsabilidades. Un buen diseño es uno donde:

  • Cada clase tiene responsabilidades claras y bien definidas
  • Las responsabilidades están relacionadas entre sí (alta cohesión)
  • Las dependencias entre clases son mínimas (bajo acoplamiento)
  • Es fácil entender, mantener y extender el código

Los Patrones GRASP

GRASP consta de 9 patrones. Hoy veremos los primeros 4:

Patrón Pregunta Clave Respuesta
1. Expert ¿A quién le asigno esta responsabilidad? A la clase que tiene la información necesaria
2. Creator ¿Quién debe crear instancias de esta clase? La clase que la contiene o la que la usa
3. Low Coupling ¿Cómo minimizo dependencias? Reduciendo referencias directas entre clases
4. High Cohesion ¿Cómo mantengo responsabilidades relacionadas? Agrupando funcionalidades afines en una clase

1. Expert (Experto)

Definición

Asigna la responsabilidad a la clase que tiene la información necesaria para cumplirla.

El Problema

Cuando necesitas implementar una funcionalidad, ¿a qué clase le asignas esa responsabilidad? Hay varias opciones, pero ¿cuál es la correcta?

Ejemplo: Calcular el Total de una Factura

Una factura tiene líneas, y cada línea tiene un precio y una cantidad. ¿Quién debería calcular el total?

  • ¿La clase Linea? Ella sabe el precio y cantidad de UNA línea, pero no de todas.
  • ¿La clase Factura? Ella sabe TODAS las líneas, así que es el experto.

Respuesta: La responsabilidad debe estar en Factura.

Aplicación en LogiSmart

Supongamos que queremos calcular la distancia total de una ruta:

¿Quién es el Experto?


// Una ruta tiene múltiples puntos de entrega
public class Ruta {
    private List puntos;
    
    // Ruta es el experto en la distancia total
    public double calcularDistanciaTotal() {
        double total = 0;
        for (int i = 0; i < puntos.size() - 1; i++) {
            Ubicacion ubicacion1 = puntos.get(i).getUbicacion();
            Ubicacion ubicacion2 = puntos.get(i + 1).getUbicacion();
            // Delega a Ubicacion (el experto en distancias)
            total += ubicacion1.calcularDistancia(ubicacion2);
        }
        return total;
    }
}

// Ubicacion es el experto en calcular distancias
public class Ubicacion {
    private double latitud;
    private double longitud;
    
    public double calcularDistancia(Ubicacion otra) {
        // Algoritmo de cálculo de distancia
        double latDiff = Math.abs(this.latitud - otra.latitud);
        double lonDiff = Math.abs(this.longitud - otra.longitud);
        return Math.sqrt(latDiff * latDiff + lonDiff * lonDiff) * 111;
    }
}

Ventajas del Patrón Expert

  • Encapsulación: La información se mantiene donde pertenece
  • Bajo acoplamiento: Los objetos no necesitan saber detalles internos unos de otros
  • Alta cohesión: Cada clase tiene responsabilidades relacionadas
  • Reutilización: Las clases son más independientes
  • Mantenibilidad: Los cambios se localizan en un lugar
💡 Regla de Oro: Si necesitas información de múltiples fuentes para hacer algo, la responsabilidad debería estar en la clase que tiene acceso a esa información.

2. Creator (Creador)

Definición

Asigna la responsabilidad de crear instancias de una clase a otra clase que tiene una relación especial con ella.

El Problema

¿Quién debe crear los objetos en tu sistema? Si no lo piensas bien, terminarás con creación inconsistente de objetos y código confuso.

Las Reglas de Creator

Creator proporciona reglas en orden de preferencia:

Regla Descripción Ejemplo en LogiSmart
1. Contención Si B contiene A (composición), B crea A Ruta contiene PuntosDeEntrega, así que Ruta los crea
2. Agregación Si B agrega A (agregación), B crea A Viaje agrega Rutas, así que Viaje las crea
3. Datos Si B tiene los datos para inicializar A, B crea A FormularioDeEnvio tiene datos del envío, puede crearlo
4. Uso Frecuente Si B usa A frecuentemente, B crea A ServicioDeNotificaciones usa Notificaciones, las crea

Aplicación en LogiSmart

Ejemplo: Quién Crea PuntoDeEntrega


// Ruta contiene PuntosDeEntrega (Regla 1: Contención)
public class Ruta {
    private List puntos;
    private Vehiculo vehiculo;
    private Operador operador;
    
    // Ruta es responsable de crear PuntosDeEntrega
    public boolean agregarPuntoDeEntrega(Envio envio, Ubicacion ubicacion, int secuencia) {
        // Validar capacidad
        if (!vehiculo.puedeTransportar(envio.getPeso())) {
            return false;
        }
        
        // Crear el punto (Ruta es el creador)
        PuntoDeEntrega punto = new PuntoDeEntrega(envio, ubicacion, secuencia);
        puntos.add(punto);
        recalcularDistancia();
        return true;
    }
}

// PuntoDeEntrega es creado por Ruta
public class PuntoDeEntrega {
    private Envio envio;
    private Ubicacion ubicacion;
    private int secuencia;
    
    public PuntoDeEntrega(Envio envio, Ubicacion ubicacion, int secuencia) {
        this.envio = envio;
        this.ubicacion = ubicacion;
        this.secuencia = secuencia;
    }
}

Ventajas del Patrón Creator

  • Consistencia: Todos los objetos se crean de la misma forma
  • Bajo acoplamiento: Los creadores no dependen de detalles complejos
  • Responsabilidad clara: Está claro quién es responsable de crear qué
  • Facilita cambios: Si el proceso de creación cambia, solo hay que modificar el creador
  • Encapsulación: El creador puede validar antes de crear
💡 Regla de Oro: La clase que contiene a otra (composición) es usualmente responsable de crearla.

3. Low Coupling (Bajo Acoplamiento)

Definición

Minimiza las dependencias entre clases.

El Problema: Alto Acoplamiento

El acoplamiento es el grado de dependencia entre dos clases. Si la clase A depende de la clase B, están acopladas.

El problema con el acoplamiento alto es que los cambios en una clase afectan a todas las que dependen de ella. Es como un efecto dominó.

❌ Diseño Incorrecto (Alto Acoplamiento)


public class Ruta {
    private Vehiculo vehiculo;
    
    // ¡Ruta crea directamente Vehiculo!
    // Si el constructor de Vehiculo cambia, hay que modificar Ruta
    public Ruta(String vehiculoId) {
        this.vehiculo = new Vehiculo(vehiculoId);
    }
}

Problema: Ruta está fuertemente acoplada a Vehiculo. Si Vehiculo cambia, Ruta se rompe.

La Solución: Inyección de Dependencias

La forma principal de lograr Low Coupling es mediante inyección de dependencias: pasar las dependencias como parámetros en lugar de crearlas internamente.

✅ Diseño Correcto (Bajo Acoplamiento)


public class Ruta {
    private Vehiculo vehiculo;
    
    // ¡Ruta recibe Vehiculo como parámetro!
    // No depende de cómo se crea Vehiculo
    public Ruta(Vehiculo vehiculo) {
        this.vehiculo = vehiculo;
    }
}

// Uso
Vehiculo vehiculo = new Vehiculo(vehiculoId);
Ruta ruta = new Ruta(vehiculo);  // Inyectamos la dependencia

Ventaja: Ruta no depende de cómo se crea Vehiculo. Es independiente.

Beneficios del Bajo Acoplamiento

  • Flexibilidad: Puedes cambiar implementaciones sin afectar otras clases
  • Testabilidad: Puedes crear mocks o stubs para testing
  • Reutilización: Las clases son más independientes
  • Mantenibilidad: Los cambios tienen menos efecto dominó
  • Escalabilidad: Es más fácil agregar nuevas funcionalidades
💡 Técnica Clave: Inyección de Dependencias. Pasa las dependencias como parámetros en lugar de crearlas internamente.

4. High Cohesion (Alta Cohesión)

Definición

Mantén las responsabilidades relacionadas juntas. Separa las no relacionadas.

El Problema: Baja Cohesión

La cohesión es el grado en que las responsabilidades de una clase están relacionadas entre sí. Si una clase hace muchas cosas no relacionadas, tiene baja cohesión.

❌ Diseño Incorrecto (Baja Cohesión)


public class Envio {
    // Atributos de dominio
    private String id;
    private Cliente cliente;
    private Ubicacion origen;
    private Ubicacion destino;
    
    // Métodos de dominio
    public double calcularCosto() { ... }
    public void marcarComoEntregado() { ... }
    
    // ¡Métodos de persistencia (no relacionados!)
    public void guardarEnBaseDatos() { ... }
    public void cargarDeBaseDatos(String id) { ... }
    
    // ¡Métodos de notificación (no relacionados!)
    public void enviarNotificacionAlCliente() { ... }
    public void registrarEnLog() { ... }
}

Problema: Envio mezcla lógica de negocio, persistencia y notificaciones. Baja cohesión.

La Solución: Separar Responsabilidades

Crea clases separadas para cada responsabilidad no relacionada:

✅ Diseño Correcto (Alta Cohesión)


// Clase de dominio: solo lógica de negocio
public class Envio {
    private String id;
    private Cliente cliente;
    private Ubicacion origen;
    private Ubicacion destino;
    
    public double calcularCosto() { ... }
    public void marcarComoEntregado() { ... }
}

// Clase para persistencia
public class EnvioRepository {
    public void guardar(Envio envio) { ... }
    public Envio obtenerPorId(String id) { ... }
}

// Clase para notificaciones
public class NotificadorDeEnvios {
    public void notificarAlCliente(Envio envio) { ... }
}

// Clase para logging
public class LoggerDeEnvios {
    public void registrar(Envio envio, String evento) { ... }
}

Beneficios de la Alta Cohesión

  • Claridad: Es fácil entender qué hace cada clase
  • Mantenibilidad: Los cambios en una responsabilidad no afectan otras
  • Reutilización: Una clase cohesiva es más fácil de reutilizar
  • Testabilidad: Es más fácil testear una clase con una responsabilidad clara
  • Escalabilidad: Es más fácil agregar nuevas responsabilidades
💡 Pregunta Clave: ¿Tiene sentido que esta clase haga esto? Si la respuesta es no, probablemente debería estar en otra clase.

5. Diagramas de Interacción

¿Qué Son?

Los diagramas de interacción visualizan cómo colaboran los objetos en el tiempo. Muestran:

  • Qué objetos participan
  • En qué orden se comunican
  • Qué mensajes se envían entre ellos
  • Qué valores retornan

Tipos de Diagramas de Interacción

Tipo Descripción Cuándo Usar
Diagrama de Secuencia Muestra el tiempo en el eje vertical (línea de tiempo) Para entender el flujo temporal de interacciones
Diagrama de Colaboración Muestra las relaciones entre objetos (mapa de comunicaciones) Para entender la estructura de relaciones

Ejemplo: Crear una Ruta en LogiSmart

Veamos un diagrama de secuencia para el caso de uso "Crear una Ruta":


Actor          UI              Sistema         Vehiculo        Ruta
  |             |                |               |              |
  |--Crear Ruta-|                |               |              |
  |             |--crearRuta---->|               |              |
  |             |                |--obtener----->|              |
  |             |                |<--Vehiculo----|              |
  |             |                |--new Ruta------------------->|
  |             |                |               |<--crear------|
  |             |                |<--Ruta--------|              |
  |             |<--Ruta---------|               |              |
  |<--Mostrar---|                |               |              |

Cómo Leer un Diagrama de Secuencia

  1. Actores/Objetos: En la parte superior, de izquierda a derecha
  2. Líneas de vida: Líneas verticales que representan la existencia del objeto
  3. Mensajes: Flechas que van de un objeto a otro (arriba hacia abajo = tiempo)
  4. Retornos: Flechas punteadas que muestran valores retornados
  5. Activaciones: Rectángulos que muestran cuándo el objeto está activo

Ventajas de los Diagramas de Interacción

  • Visualización: Es más fácil entender el flujo que leyendo código
  • Comunicación: Puedes explicar el diseño a otros de forma clara
  • Validación: Puedes verificar que los objetos se comunican correctamente
  • Identificación de problemas: Puedes ver si hay acoplamiento excesivo o responsabilidades mal asignadas
  • Documentación: Es una forma de documentar el comportamiento del sistema
💡 Consejo: Dibuja diagramas de secuencia para los casos de uso más importantes. Te ayudarán a validar tu diseño y comunicarlo a otros.

Resumen: GRASP Parte I

Los 4 Patrones

Patrón Pregunta Respuesta
Expert ¿A quién le asigno esta responsabilidad? A la clase que tiene la información necesaria
Creator ¿Quién debe crear instancias? La clase que la contiene o la que la usa
Low Coupling ¿Cómo minimizo dependencias? Inyectando dependencias, no creándolas internamente
High Cohesion ¿Cómo mantengo responsabilidades relacionadas? Separando responsabilidades no relacionadas en otras clases

Cómo Aplicar GRASP

Cuando diseñes, hazte estas preguntas:

  1. Expert: ¿Quién es el experto en esta información?
  2. Creator: ¿Quién debería crear este objeto?
  3. Low Coupling: ¿Cómo puedo reducir dependencias?
  4. High Cohesion: ¿Tiene sentido que esta clase haga esto?
  5. Diagramas: ¿Cómo colaboran los objetos? Dibuja un diagrama.
