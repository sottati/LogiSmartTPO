Objetivo de la Clase

Completar el estudio de patrones estructurales con cuatro patrones avanzados que resuelven problemas específicos:

  • Agregar responsabilidades dinámicamente a objetos
  • Simplificar interfaces complejas
  • Optimizar memoria compartiendo objetos
  • Controlar acceso a objetos costosos

🎨 Decorator Pattern

Problema: Explosión de Subclases

Tienes una clase `Envio` y necesitas agregar funcionalidades opcionales:

  • Envío con seguro
  • Envío con rastreo GPS
  • Envío con seguro Y rastreo
  • Envío con seguro Y rastreo Y notificaciones

Solución ingenua con herencia:

Envio ├── EnvioConSeguro ├── EnvioConRastreo ├── EnvioConSeguroYRastreo ├── EnvioConNotificaciones ├── EnvioConSeguroYNotificaciones ├── EnvioConRastreoYNotificaciones └── EnvioConSeguroYRastreoYNotificaciones Total: 8 clases (2^3 combinaciones) Si agregas otra funcionalidad: 16 clases Si agregas otra: 32 clases

Problema: Explosión combinatoria. Es imposible mantener.

Solución: Decorator

Envuelve objetos con decoradores que agregan funcionalidad sin modificar el objeto original.

┌─────────────────────────────────┐ │ Envio Original │ │ - obtenerCosto() │ │ - obtenerTiempo() │ └─────────────────────────────────┘ ▲ │ Envuelto por ▼ ┌─────────────────────────────────┐ │ DecoradorSeguro │ │ - obtenerCosto() + seguro │ │ - obtenerTiempo() │ └─────────────────────────────────┘ ▲ │ Envuelto por ▼ ┌─────────────────────────────────┐ │ DecoradorRastreo │ │ - obtenerCosto() + rastreo │ │ - obtenerTiempo() + GPS │ └─────────────────────────────────┘ Resultado: 1 clase + 3 decoradores = 4 clases Mucho mejor que 8 clases

Implementación

Paso 1: Interfaz Base

public interface Envio {
    double obtenerCosto();
    int obtenerTiempoEntrega(); // en horas
    String obtenerDescripcion();
}

public class EnvioBasico implements Envio {
    private String origen;
    private String destino;
    private double peso;
    
    public EnvioBasico(String origen, String destino, double peso) {
        this.origen = origen;
        this.destino = destino;
        this.peso = peso;
    }
    
    @Override
    public double obtenerCosto() {
        return peso * 10.0; // $10 por kg
    }
    
    @Override
    public int obtenerTiempoEntrega() {
        return 3; // 3 días
    }
    
    @Override
    public String obtenerDescripcion() {
        return "Envío básico: " + origen + " -> " + destino;
    }
}

Paso 2: Decorador Base Abstracto

public abstract class DecoradorEnvio implements Envio {
    protected Envio envioDecorado;
    
    public DecoradorEnvio(Envio envio) {
        this.envioDecorado = envio;
    }
    
    @Override
    public double obtenerCosto() {
        return envioDecorado.obtenerCosto();
    }
    
    @Override
    public int obtenerTiempoEntrega() {
        return envioDecorado.obtenerTiempoEntrega();
    }
    
    @Override
    public String obtenerDescripcion() {
        return envioDecorado.obtenerDescripcion();
    }
}

Paso 3: Decoradores Concretos

// Decorador: Agregar Seguro
public class DecoradorSeguro extends DecoradorEnvio {
    public DecoradorSeguro(Envio envio) {
        super(envio);
    }
    
    @Override
    public double obtenerCosto() {
        return envioDecorado.obtenerCosto() * 1.15; // +15%
    }
    
    @Override
    public String obtenerDescripcion() {
        return envioDecorado.obtenerDescripcion() + " + Seguro";
    }
}

// Decorador: Agregar Rastreo GPS
public class DecoradorRastreoGPS extends DecoradorEnvio {
    public DecoradorRastreoGPS(Envio envio) {
        super(envio);
    }
    
    @Override
    public double obtenerCosto() {
        return envioDecorado.obtenerCosto() + 50.0; // +$50
    }
    
    @Override
    public int obtenerTiempoEntrega() {
        return envioDecorado.obtenerTiempoEntrega() - 1; // 1 día menos
    }
    
    @Override
    public String obtenerDescripcion() {
        return envioDecorado.obtenerDescripcion() + " + Rastreo GPS";
    }
}

// Decorador: Agregar Notificaciones
public class DecoradorNotificaciones extends DecoradorEnvio {
    public DecoradorNotificaciones(Envio envio) {
        super(envio);
    }
    
    @Override
    public double obtenerCosto() {
        return envioDecorado.obtenerCosto() + 25.0; // +$25
    }
    
    @Override
    public String obtenerDescripcion() {
        return envioDecorado.obtenerDescripcion() + " + Notificaciones";
    }
}

Paso 4: Uso del Decorator

// Crear envío básico
Envio envio = new EnvioBasico("Buenos Aires", "Córdoba", 5.0);
System.out.println(envio.obtenerDescripcion()); // Envío básico...
System.out.println("Costo: $" + envio.obtenerCosto()); // $50

// Agregar seguro
envio = new DecoradorSeguro(envio);
System.out.println(envio.obtenerDescripcion()); // ...+ Seguro
System.out.println("Costo: $" + envio.obtenerCosto()); // $57.50

// Agregar rastreo
envio = new DecoradorRastreoGPS(envio);
System.out.println(envio.obtenerDescripcion()); // ...+ Rastreo GPS
System.out.println("Costo: $" + envio.obtenerCosto()); // $107.50

// Agregar notificaciones
envio = new DecoradorNotificaciones(envio);
System.out.println(envio.obtenerDescripcion()); // ...+ Notificaciones
System.out.println("Costo: $" + envio.obtenerCosto()); // $132.50

Ventajas del Decorator

  • Flexibilidad: Combina decoradores en cualquier orden
  • Evita Explosión de Clases: 4 clases en lugar de 8
  • Responsabilidad Única: Cada decorador hace una cosa
  • Composición sobre Herencia: Más flexible que subclases

Desventajas

  • Complejidad: Muchas clases pequeñas
  • Orden Importante: El orden de decoradores puede importar
  • Debugging Difícil: Stack de decoradores es complejo

Cuándo Usar Decorator

  • Necesitas agregar responsabilidades dinámicamente
  • Quieres evitar explosión de subclases
  • Las responsabilidades son opcionales e independientes
  • El orden de aplicación importa

🏛️ Facade Pattern

Problema: Complejidad Oculta

Tu sistema de logística tiene muchos subsistemas complejos:

  • Sistema de Inventario
  • Sistema de Pagos
  • Sistema de Notificaciones
  • Sistema de Reportes
  • Sistema de Rastreo

Para crear un envío, el cliente debe:

// Código sin Facade (complejo)
Inventario inventario = new Inventario();
inventario.verificarStock("PROD-001");
inventario.restarStock("PROD-001", 1);

Pago pago = new Pago();
pago.procesarPago("VISA", 150.0);

Notificacion notif = new Notificacion();
notif.enviarEmail("cliente@email.com", "Pedido confirmado");

Rastreo rastreo = new Rastreo();
rastreo.crearNumeroSeguimiento();

Reporte reporte = new Reporte();
reporte.generarReporte("ENVIO", "PDF");

Problema: El cliente necesita conocer todos los subsistemas y su orden.

Solución: Facade

Crea una interfaz unificada que encapsula la complejidad.

┌─────────────────────────────────┐ │ Cliente │ └─────────────────────────────────┘ │ │ Usa ▼ ┌─────────────────────────────────┐ │ ServicioEnviosFacade │ │ - crearEnvio(datos) │ │ - cancelarEnvio(id) │ │ - obtenerEstado(id) │ └─────────────────────────────────┘ │ │ Coordina ├─────────────────────────────────┐ ▼ ▼ ┌─────────────┐ ┌──────────────────┐ │ Inventario │ │ Sistema de Pagos │ └─────────────┘ └──────────────────┘ │ │ ├─────────────────────────────────┤ ▼ ▼ ┌──────────────┐ ┌──────────────────┐ │Notificaciones│ │ Sistema de Rastreo └──────────────┘ └──────────────────┘

Implementación

Paso 1: Subsistemas Complejos

public class SistemaInventario {
    public boolean verificarStock(String productoId) {
        System.out.println("[Inventario] Verificando stock de " + productoId);
        return true;
    }
    
    public void restarStock(String productoId, int cantidad) {
        System.out.println("[Inventario] Restando " + cantidad + " de " + productoId);
    }
}

public class SistemaPagos {
    public boolean procesarPago(String metodo, double monto) {
        System.out.println("[Pagos] Procesando $" + monto + " con " + metodo);
        return true;
    }
}

public class SistemaNotificaciones {
    public void enviarEmail(String email, String mensaje) {
        System.out.println("[Notificaciones] Email a " + email + ": " + mensaje);
    }
    
    public void enviarSMS(String telefono, String mensaje) {
        System.out.println("[Notificaciones] SMS a " + telefono + ": " + mensaje);
    }
}

public class SistemaRastreo {
    public String crearNumeroSeguimiento() {
        String numero = "TRK-" + System.currentTimeMillis();
        System.out.println("[Rastreo] Número de seguimiento: " + numero);
        return numero;
    }
}

Paso 2: Facade que Encapsula

public class ServicioEnviosFacade {
    private SistemaInventario inventario = new SistemaInventario();
    private SistemaPagos pagos = new SistemaPagos();
    private SistemaNotificaciones notificaciones = new SistemaNotificaciones();
    private SistemaRastreo rastreo = new SistemaRastreo();
    
    /**
     * Crea un envío coordinando todos los subsistemas
     */
    public String crearEnvio(String productoId, double monto, String email) {
        System.out.println("\n=== Creando Envío ===");
        
        // 1. Verificar inventario
        if (!inventario.verificarStock(productoId)) {
            System.err.println("Stock insuficiente");
            return null;
        }
        
        // 2. Procesar pago
        if (!pagos.procesarPago("TARJETA", monto)) {
            System.err.println("Pago rechazado");
            return null;
        }
        
        // 3. Restar stock
        inventario.restarStock(productoId, 1);
        
        // 4. Crear número de seguimiento
        String numeroSeguimiento = rastreo.crearNumeroSeguimiento();
        
        // 5. Notificar al cliente
        notificaciones.enviarEmail(email, 
            "Tu envío ha sido confirmado. Número: " + numeroSeguimiento);
        
        System.out.println("✓ Envío creado exitosamente\n");
        return numeroSeguimiento;
    }
    
    /**
     * Cancela un envío
     */
    public void cancelarEnvio(String numeroSeguimiento) {
        System.out.println("\n=== Cancelando Envío ===");
        System.out.println("[Rastreo] Cancelando " + numeroSeguimiento);
        System.out.println("[Pagos] Reembolsando pago");
        System.out.println("[Notificaciones] Notificando cancelación");
        System.out.println("✓ Envío cancelado\n");
    }
}

Paso 3: Uso del Facade

// Código con Facade (simple)
ServicioEnviosFacade servicio = new ServicioEnviosFacade();

String numeroSeguimiento = servicio.crearEnvio(
    "PROD-001",
    150.0,
    "cliente@email.com"
);

servicio.cancelarEnvio(numeroSeguimiento);

Ventajas del Facade

  • Simplificación: Interfaz simple para complejidad
  • Desacoplamiento: Cliente no conoce subsistemas
  • Mantenibilidad: Cambios internos no afectan cliente
  • Reutilización: Múltiples clientes usan la misma fachada

Desventajas

  • Dios Object: Facade puede volverse muy grande
  • Oculta Complejidad: A veces necesitas acceso directo
  • Menos Flexible: Limitado a operaciones predefinidas

Cuándo Usar Facade

  • Tienes subsistemas complejos que interactúan
  • Quieres simplificar la interfaz para clientes
  • Necesitas desacoplar cliente de subsistemas
  • Hay operaciones comunes que coordinan varios subsistemas

💾 Flyweight Pattern

Problema: Consumo Excesivo de Memoria

Tu sistema de logística crea millones de objetos `Ubicacion`:

// Sin Flyweight (ineficiente)
List ubicaciones = new ArrayList<>();
for (int i = 0; i < 1_000_000; i++) {
    ubicaciones.add(new Ubicacion("Buenos Aires", "Buenos Aires", "1425"));
    ubicaciones.add(new Ubicacion("Córdoba", "Córdoba", "5000"));
    ubicaciones.add(new Ubicacion("Rosario", "Santa Fe", "2000"));
}
// Resultado: 3 millones de objetos idénticos en memoria
// Cada uno ocupa ~100 bytes = 300 MB de memoria desperdiciada

Problema: Muchos objetos idénticos consumiendo memoria innecesariamente.

Solución: Flyweight

Comparte objetos inmutables para reducir consumo de memoria.

Sin Flyweight: Ubicacion("Buenos Aires") → Objeto 1 (100 bytes) Ubicacion("Buenos Aires") → Objeto 2 (100 bytes) Ubicacion("Buenos Aires") → Objeto 3 (100 bytes) ... Total: 1,000,000 objetos × 100 bytes = 100 MB Con Flyweight: Ubicacion("Buenos Aires") → Objeto Compartido (100 bytes) Ubicacion("Buenos Aires") → Referencia al Objeto Compartido Ubicacion("Buenos Aires") → Referencia al Objeto Compartido ... Total: 1 objeto × 100 bytes = 100 bytes

Implementación

Paso 1: Objeto Flyweight (Inmutable)

public class Ubicacion {
    private final String ciudad;
    private final String provincia;
    private final String codigoPostal;
    
    public Ubicacion(String ciudad, String provincia, String codigoPostal) {
        this.ciudad = ciudad;
        this.provincia = provincia;
        this.codigoPostal = codigoPostal;
    }
    
    public String getCiudad() { return ciudad; }
    public String getProvincia() { return provincia; }
    public String getCodigoPostal() { return codigoPostal; }
    
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Ubicacion ubicacion = (Ubicacion) o;
        return ciudad.equals(ubicacion.ciudad) &&
               provincia.equals(ubicacion.provincia) &&
               codigoPostal.equals(ubicacion.codigoPostal);
    }
    
    @Override
    public int hashCode() {
        return java.util.Objects.hash(ciudad, provincia, codigoPostal);
    }
}

Paso 2: Factory de Flyweights

public class FabricaUbicaciones {
    private static final Map ubicaciones = new HashMap<>();
    
    /**
     * Obtiene o crea una ubicación (compartida)
     */
    public static Ubicacion obtener(String ciudad, String provincia, String codigoPostal) {
        String clave = ciudad + "|" + provincia + "|" + codigoPostal;
        
        if (!ubicaciones.containsKey(clave)) {
            Ubicacion nueva = new Ubicacion(ciudad, provincia, codigoPostal);
            ubicaciones.put(clave, nueva);
            System.out.println("[Flyweight] Nueva ubicación creada: " + clave);
        } else {
            System.out.println("[Flyweight] Reutilizando ubicación: " + clave);
        }
        
        return ubicaciones.get(clave);
    }
    
    /**
     * Obtiene estadísticas
     */
    public static void mostrarEstadisticas() {
        System.out.println("Ubicaciones únicas en memoria: " + ubicaciones.size());
    }
}

Paso 3: Uso del Flyweight

// Sin Flyweight
List ubicacionesSin = new ArrayList<>();
for (int i = 0; i < 1_000_000; i++) {
    ubicacionesSin.add(new Ubicacion("Buenos Aires", "Buenos Aires", "1425"));
}
// 1 millón de objetos en memoria

// Con Flyweight
List ubicacionesCon = new ArrayList<>();
for (int i = 0; i < 1_000_000; i++) {
    ubicacionesCon.add(FabricaUbicaciones.obtener("Buenos Aires", "Buenos Aires", "1425"));
}
// Solo 1 objeto en memoria, 1 millón de referencias

FabricaUbicaciones.mostrarEstadisticas();
// Output: Ubicaciones únicas en memoria: 1

Ventajas del Flyweight

  • Optimización de Memoria: Reduce consumo dramáticamente
  • Rendimiento: Menos objetos = menos garbage collection
  • Escalabilidad: Soporta millones de objetos

Desventajas

  • Complejidad: Requiere factory y gestión de pool
  • Thread-Safety: Factory debe ser thread-safe
  • Inmutabilidad Requerida: Los objetos no pueden cambiar

Cuándo Usar Flyweight

  • Creas muchos objetos similares
  • El consumo de memoria es crítico
  • Los objetos son inmutables
  • Hay mucha reutilización potencial

Ejemplos en Java Estándar

  • String.intern() - Comparte strings idénticos
  • Integer.valueOf() - Cache de enteros pequeños (-128 a 127)
  • Boolean.TRUE / Boolean.FALSE - Objetos únicos compartidos

🔐 Proxy Pattern

Problema: Acceso Costoso

Tu sistema necesita acceder a recursos costosos:

  • Conexión a base de datos remota
  • Descarga de archivo grande
  • Cálculos complejos
  • Acceso a servicios externos

Problema: Cargar todos los recursos al inicio es ineficiente.

Solución: Proxy

Crea un proxy que controla el acceso al recurso real (lazy loading, caché, validación).

┌─────────────────────────────────┐ │ Cliente │ └─────────────────────────────────┘ │ │ Usa ▼ ┌─────────────────────────────────┐ │ ProxyBaseDatos │ │ - Lazy loading │ │ - Caché │ │ - Validación │ └─────────────────────────────────┘ │ │ Accede cuando es necesario ▼ ┌─────────────────────────────────┐ │ BaseDatosReal │ │ - Conexión costosa │ │ - Consultas complejas │ └─────────────────────────────────┘

Tipos de Proxy

  • Proxy Virtual: Lazy loading (carga bajo demanda)
  • Proxy de Protección: Control de acceso (validación)
  • Proxy Remoto: Acceso a objetos remotos
  • Proxy de Caché: Almacena resultados

Implementación: Proxy Virtual

Paso 1: Interfaz Común

public interface BaseDatos {
    List obtenerEnvios();
    Envio obtenerEnvio(String id);
    void guardarEnvio(Envio envio);
}

public class BaseDatosReal implements BaseDatos {
    public BaseDatosReal() {
        System.out.println("[BD Real] Conectando a servidor remoto...");
        // Simular conexión costosa
        try { Thread.sleep(2000); } catch (InterruptedException e) {}
        System.out.println("[BD Real] Conexión establecida");
    }
    
    @Override
    public List obtenerEnvios() {
        System.out.println("[BD Real] Consultando envíos...");
        return new ArrayList<>();
    }
    
    @Override
    public Envio obtenerEnvio(String id) {
        System.out.println("[BD Real] Consultando envío: " + id);
        return new Envio(...);
    }
    
    @Override
    public void guardarEnvio(Envio envio) {
        System.out.println("[BD Real] Guardando envío");
    }
}

Paso 2: Proxy Virtual (Lazy Loading)

public class ProxyBaseDatos implements BaseDatos {
    private BaseDatosReal bdReal;
    private Map cache = new HashMap<>();
    
    /**
     * Carga la BD solo cuando es necesario
     */
    private BaseDatosReal obtenerBD() {
        if (bdReal == null) {
            System.out.println("[Proxy] Inicializando BD...");
            bdReal = new BaseDatosReal();
        }
        return bdReal;
    }
    
    @Override
    public List obtenerEnvios() {
        System.out.println("[Proxy] Obteniendo envíos");
        return obtenerBD().obtenerEnvios();
    }
    
    @Override
    public Envio obtenerEnvio(String id) {
        System.out.println("[Proxy] Buscando en caché: " + id);
        
        if (cache.containsKey(id)) {
            System.out.println("[Proxy] ✓ Encontrado en caché");
            return cache.get(id);
        }
        
        System.out.println("[Proxy] ✗ No en caché, consultando BD");
        Envio envio = obtenerBD().obtenerEnvio(id);
        cache.put(id, envio);
        return envio;
    }
    
    @Override
    public void guardarEnvio(Envio envio) {
        System.out.println("[Proxy] Guardando y actualizando caché");
        obtenerBD().guardarEnvio(envio);
        cache.put(envio.getId(), envio);
    }
}

Paso 3: Uso del Proxy

// Crear proxy (BD real NO se conecta aún)
BaseDatos bd = new ProxyBaseDatos();
System.out.println("Proxy creado, BD no conectada");

// Primera consulta (conecta la BD)
Envio envio1 = bd.obtenerEnvio("ENV-001");

// Segunda consulta (usa caché, no consulta BD)
Envio envio2 = bd.obtenerEnvio("ENV-001");

Ventajas del Proxy

  • Lazy Loading: Carga recursos solo cuando se necesitan
  • Caché: Mejora rendimiento reutilizando resultados
  • Control de Acceso: Valida permisos antes de acceder
  • Transparencia: Cliente usa proxy como si fuera el objeto real

Desventajas

  • Complejidad: Capa adicional de indirección
  • Rendimiento: Pequeña sobrecarga en cada acceso
  • Sincronización: Caché puede quedar desactualizado

Cuándo Usar Proxy

  • Acceso a recursos costosos (BD, red, cálculos)
  • Necesitas lazy loading
  • Quieres implementar caché
  • Necesitas control de acceso

📊 Comparación de Patrones

Tabla Comparativa

Patrón Propósito Complejidad Cuándo Usar
Decorator Agregar responsabilidades dinámicamente Media Funcionalidades opcionales e independientes
Facade Simplificar interfaces complejas Baja Subsistemas complejos con operaciones comunes
Flyweight Optimizar memoria compartiendo objetos Media Muchos objetos similares, memoria crítica
Proxy Controlar acceso a recursos costosos Media Lazy loading, caché, validación

Matriz de Decisión

  • ¿Agregar funcionalidad dinámicamente? → Decorator
  • ¿Simplificar subsistemas complejos? → Facade
  • ¿Optimizar memoria? → Flyweight
  • ¿Controlar acceso a recursos? → Proxy

Patrón Categoría Propósito principal
Adapter Estructural Compatibilidad de interfaces: permite que clases con interfaces incompatibles trabajen juntas convirtiendo una interfaz en otra [1] [2]
Bridge Estructural Separar abstracción e implementación: divide una clase en dos jerarquías independientes (abstracción y implementación) para poder cambiarlas por separado [6] [9]
Composite Estructural Estructuras jerárquicas: compone objetos en estructuras de árbol para representar relaciones parte-todo, permitiendo tratar objetos individuales y compuestos uniformemente [7] [10]
Decorator Estructural Agregar responsabilidades: añade funcionalidad dinámicamente a un objeto sin modificar su código, mediante composición recursiva [1]
Facade Estructural Simplificar complejidad: crea una interfaz simplificada para acceder a un subsistema complejo [2] [4]
Flyweight Estructural Optimizar memoria: comparte información común entre múltiples objetos para reducir el uso de memoria [10]
Proxy Estructural Controlar acceso: actúa como representante o intermediario de un objeto para controlar su acceso [4]

Nota importante: Todos estos patrones son estructurales, no creacionales. Los patrones creacionales (como Singleton, Factory, Abstract Factory, Builder, Prototype) se enfocan en cómo se crean los objetos, mientras que estos patrones estructurales se enfocan en cómo se componen las clases y objetos para formar estructuras más grandes [1] [2].

🔗 Combinación de Patrones

Decorator + Proxy

Combina lazy loading con funcionalidades opcionales:

// Proxy carga la BD bajo demanda
BaseDatos bd = new ProxyBaseDatos();

// Decorator agrega caché adicional
BaseDatos bdConCache = new DecoradorCacheAvanzado(bd);

// Decorator agrega logging
BaseDatos bdConLogging = new DecoradorLogging(bdConCache);

// El cliente usa la interfaz simple
Envio envio = bdConLogging.obtenerEnvio("ENV-001");

Facade + Decorator

Simplifica subsistemas pero permite agregar funcionalidad:

// Facade simplifica
ServicioEnviosFacade servicio = new ServicioEnviosFacade();

// Decorator agrega auditoría
ServicioEnviosFacade servicioAuditado = 
    new DecoradorAuditoria(servicio);

// Decorator agrega notificaciones
ServicioEnviosFacade servicioNotificado = 
    new DecoradorNotificaciones(servicioAuditado);

// Usar
servicioNotificado.crearEnvio(...);

Composite + Flyweight

Estructura jerárquica con objetos compartidos:

// Flyweight: ubicaciones compartidas
Ubicacion buenosAires = FabricaUbicaciones.obtener(
    "Buenos Aires", "Buenos Aires", "1425"
);

// Composite: jerarquía de centros
CentroRegional argentina = new CentroRegional(...);
PuntoEntrega punto1 = new PuntoEntrega(..., buenosAires);
PuntoEntrega punto2 = new PuntoEntrega(..., buenosAires);

// Ambos puntos comparten la misma ubicación
argentina.agregar(punto1);
argentina.agregar(punto2);

Adapter + Facade

Integra componentes externos y simplifica acceso:

// Adapter: integra proveedores
ProveedorEnvio dhl = new AdapterDHL();
ProveedorEnvio fedex = new AdapterFedEx();

// Facade: simplifica selección de proveedor
ServicioProveedoresFacade servicio = 
    new ServicioProveedoresFacade(dhl, fedex);

// Cliente usa interfaz simple
servicio.crearEnvio("DHL", envio);

Principios de Combinación

  • Composición sobre Herencia: Combina patrones en lugar de crear subclases
  • Responsabilidad Única: Cada patrón resuelve un problema específico
  • Interfaz Consistente: Los patrones combinados mantienen interfaz clara
  • Evitar Complejidad Excesiva: No combines más de 3-4 patrones

💼 Casos de Aplicación en Sistemas Reales

Decorator en Sistemas Reales

Caso 1: Streams de Java

BufferedInputStream, DataInputStream son decoradores de InputStream:

InputStream stream = new FileInputStream("archivo.txt");
InputStream buffered = new BufferedInputStream(stream);
InputStream data = new DataInputStream(buffered);
// Cada uno agrega funcionalidad sin modificar el anterior

Caso 2: Componentes GUI

Agregar bordes, scrollbars, efectos a componentes sin modificar el componente base.

Caso 3: Middleware Web

Express.js usa decoradores para agregar funcionalidad a rutas (autenticación, logging, validación).

Facade en Sistemas Reales

Caso 1: Bibliotecas Gráficas

OpenGL es complejo, pero librerías como Lwjgl proporcionan facades simplificadas.

Caso 2: Frameworks Web

Spring Framework proporciona facades para JDBC, JPA, transacciones.

Caso 3: APIs Públicas

Google Maps API simplifica acceso a servicios complejos de mapas.

Flyweight en Sistemas Reales

Caso 1: Editores de Texto

Cada carácter comparte objetos de fuente, color, estilo para ahorrar memoria.

Caso 2: Videojuegos

Texturas, modelos 3D se comparten entre múltiples objetos del juego.

Caso 3: Bases de Datos

Connection pooling reutiliza conexiones en lugar de crear nuevas.

Proxy en Sistemas Reales

Caso 1: ORM (Hibernate, JPA)

Lazy loading: las relaciones se cargan solo cuando se acceden.

Caso 2: Caché Distribuido (Redis)

Proxy intercepta consultas y retorna resultados en caché si están disponibles.

Caso 3: Servicios Web

Proxy remoto permite acceder a objetos en servidores remotos como si estuvieran locales.

✅ Resumen de la Clase

Conceptos Clave

Decorator

  • Envuelve objetos para agregar funcionalidad
  • Evita explosión de subclases
  • Permite combinar decoradores en cualquier orden

Facade

  • Simplifica interfaces complejas
  • Encapsula subsistemas
  • Desacopla cliente de implementación

Flyweight

  • Comparte objetos inmutables para optimizar memoria
  • Reduce consumo dramáticamente
  • Requiere factory para gestionar pool

Proxy

  • Controla acceso a recursos costosos
  • Implementa lazy loading y caché
  • Proporciona transparencia al cliente
