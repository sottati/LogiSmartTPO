🏭 Repaso: Patrones Creacionales I (Clase 6)

En Clase 6 comenzamos con los patrones creacionales. Vimos dos patrones fundamentales:

Patrón Propósito Ejemplo en LogiSmart
Singleton Una única instancia en toda la app ConfiguracionSistema.getInstance()
Factory Method Crear objetos sin especificar clase exacta VehiculoFactory.crearVehiculo(tipo)

¿Por qué necesitamos tantos patrones creacionales?

Porque cada patrón resuelve un problema diferente:

  • Singleton: Cuando necesitas exactamente 1 instancia
  • Factory Method: Cuando necesitas crear 1 tipo de objeto, pero esconder la clase concreta
  • Abstract Factory: Cuando necesitas crear FAMILIAS de objetos relacionados (lo veremos hoy)
  • Builder: Cuando necesitas crear objetos complejos paso a paso (lo veremos hoy)
  • Prototype: Cuando necesitas clonar objetos existentes (lo veremos hoy)

🏭 Abstract Factory: Introducción

El Problema

Imagina que LogiSmart necesita funcionar en diferentes regiones: Argentina, Brasil y México. Cada región tiene:

  • Diferentes tipos de vehículos disponibles
  • Diferentes proveedores de mapas
  • Diferentes formas de calcular impuestos
  • Diferentes regulaciones

Código Incorrecto (❌)

if (region.equals("Argentina")) {
    vehiculo = new AutoArgentino();
    mapas = new GoogleMapsArgentina();
    impuestos = new ImpuestosArgentina();
} else if (region.equals("Brasil")) {
    vehiculo = new AutoBrasil();
    mapas = new HereMaps();
    impuestos = new ImpuestosBrasil();
} else if (region.equals("Mexico")) {
    vehiculo = new AutoMexico();
    mapas = new OpenStreetMap();
    impuestos = new ImpuestosMexico();
}

Problemas: Código lleno de condicionales, difícil de mantener, acoplado a clases concretas

La Solución: Abstract Factory

Abstract Factory es un patrón que permite crear familias de objetos relacionados sin especificar sus clases concretas.

Analogía

Es como tener una fábrica que produce todo lo que necesitas para una región específica. El cliente no necesita saber qué ingredientes específicos se usan; solo pide "quiero el menú argentino" y recibe todo lo que necesita.

💻 Abstract Factory: Implementación

Paso 1: Definir las Interfaces

// Interfaz que define qué objetos debe crear
public interface VehiculoFactory {
    Vehiculo crearVehiculo();
    ProveedorMapas crearProveedorMapas();
    CalculadorImpuestos crearCalculadorImpuestos();
}

// Interfaces de los productos
public interface Vehiculo { void conducir(); }
public interface ProveedorMapas { void obtenerRuta(); }
public interface CalculadorImpuestos { double calcular(double monto); }

Paso 2: Implementaciones Concretas para Argentina

public class VehiculoFactoryArgentina implements VehiculoFactory {
    @Override
    public Vehiculo crearVehiculo() {
        return new AutoArgentino();
    }

    @Override
    public ProveedorMapas crearProveedorMapas() {
        return new GoogleMapsArgentina();
    }

    @Override
    public CalculadorImpuestos crearCalculadorImpuestos() {
        return new ImpuestosArgentina();
    }
}

Paso 3: Uso en el Cliente

public class LogiSmartApp {
    private VehiculoFactory factory;

    public LogiSmartApp(String region) {
        if (region.equals("Argentina")) {
            factory = new VehiculoFactoryArgentina();
        } else if (region.equals("Brasil")) {
            factory = new VehiculoFactoryBrasil();
        }
    }

    public void iniciar() {
        Vehiculo vehiculo = factory.crearVehiculo();
        ProveedorMapas mapas = factory.crearProveedorMapas();
        CalculadorImpuestos impuestos = factory.crearCalculadorImpuestos();
    }
}

✅ Ventajas

  • Garantiza que los objetos de una familia se crean juntos
  • Desacopla el cliente de las clases concretas
  • Fácil agregar nuevas regiones (nueva factory)
  • Cambiar toda una familia de objetos es simple

🔨 Builder: Introducción

El Problema

Un Envio en LogiSmart tiene muchos atributos, muchos de ellos opcionales:

  • id (requerido)
  • origen (requerido)
  • destino (requerido)
  • descripcion (opcional)
  • peso (opcional)
  • frágil (opcional)
  • requiereSignatura (opcional)
  • requiereRefrigeración (opcional)
  • requiereAseguranza (opcional)
  • instruccionesEspeciales (opcional)
  • contactoEmergencia (opcional)
  • horaEntregaPreferida (opcional)

Código Incorrecto (❌)

// ¿Hacemos 2^11 constructores? ¡Imposible!
Envio envio = new Envio("ENV001", "Buenos Aires", "Córdoba",
                        "Medicinas", 2.5, true, true, true,
                        true, "Mantener frío", "Dr. García",
                        LocalTime.of(14, 0));

Problemas: Código ilegible, imposible mantener, fácil de equivocarse

La Solución: Builder

Builder es un patrón que permite construir objetos complejos paso a paso. Separa la construcción de la representación.

Analogía

Es como construir una casa. No haces todo de una vez. Primero los cimientos, luego las paredes, luego el techo, luego los acabados. Builder es como tener un constructor que sigue estos pasos.

💻 Builder: Implementación

Paso 1: Clase Envio con Constructor Privado

public class Envio {
    private String id;
    private String origen;
    private String destino;
    private String descripcion;
    // ... más atributos

    // Constructor privado - solo Builder puede crear Envio
    private Envio(EnvioBuilder builder) {
        this.id = builder.id;
        this.origen = builder.origen;
        // ... asignar más atributos
    }

    // Getters
    public String getId() { return id; }
    public String getOrigen() { return origen; }
    // ... más getters
}

Paso 2: Clase Builder

public class EnvioBuilder {
    private String id;
    private String origen;
    private String destino;
    private String descripcion = "";
    private double peso = 0;
    private boolean fragil = false;
    // ... más atributos

    // Constructor con parámetros requeridos
    public EnvioBuilder(String id, String origen, String destino) {
        this.id = id;
        this.origen = origen;
        this.destino = destino;
    }

    // Métodos para parámetros opcionales - retornan this
    public EnvioBuilder descripcion(String descripcion) {
        this.descripcion = descripcion;
        return this;
    }

    public EnvioBuilder peso(double peso) {
        this.peso = peso;
        return this;
    }

    public EnvioBuilder fragil(boolean fragil) {
        this.fragil = fragil;
        return this;
    }

    // ... más métodos

    // Método build() que crea el Envio
    public Envio build() {
        return new Envio(this);
    }
}

Paso 3: Uso - Código Correcto (✅)

// Envio simple
Envio envio1 = new EnvioBuilder("ENV001", "Buenos Aires", "Córdoba")
    .build();

// Envio complejo - ¡Mucho más legible!
Envio envio2 = new EnvioBuilder("ENV002", "Buenos Aires", "Mendoza")
    .descripcion("Medicinas urgentes")
    .peso(2.5)
    .fragil(true)
    .requiereSignatura(true)
    .requiereRefrigeracion(true)
    .requiereAseguranza(true)
    .instruccionesEspeciales("Mantener en frío")
    .contactoEmergencia("Dr. García: 555-1234")
    .horaEntregaPreferida(LocalTime.of(14, 0))
    .build();

✅ Ventajas

  • Código muy legible y expresivo
  • Fácil agregar nuevos atributos opcionales
  • Evita constructores con muchos parámetros
  • Garantiza que el objeto está completamente construido

🔄 Prototype: Introducción

El Problema

Tienes un Envio completamente configurado. Necesitas crear 100 envios más con la misma configuración. ¿Qué haces?

Código Incorrecto (❌)

// Llamar 100 veces al Builder - ¡Ineficiente!
for (int i = 0; i < 100; i++) {
    Envio envio = new EnvioBuilder("ENV-" + i, "Buenos Aires", "Córdoba")
        .descripcion("Medicinas")
        .peso(2.5)
        .fragil(true)
        .requiereSignatura(true)
        .requiereRefrigeracion(true)
        .requiereAseguranza(true)
        .instruccionesEspeciales("Mantener frío")
        .contactoEmergencia("Dr. García")
        .horaEntregaPreferida(LocalTime.of(14, 0))
        .build();
    envios.add(envio);
}

Problema: Ineficiente, repetitivo, lento

La Solución: Prototype

Prototype es un patrón que permite crear nuevas instancias clonando un objeto existente. Es mucho más eficiente que crear desde cero.

Analogía

Es como hacer una fotocopia. En lugar de escribir un documento desde cero 100 veces, haces una copia del original 100 veces.

💻 Prototype: Implementación

Paso 1: Implementar Cloneable

public class Envio implements Cloneable {
    private String id;
    private String origen;
    private String destino;
    // ... más atributos
    private List<String> instrucciones;

    // Implementar clone()
    @Override
    public Envio clone() {
        try {
            Envio clonado = (Envio) super.clone();
            // IMPORTANTE: Clonar también las listas (Deep Copy)
            clonado.instrucciones = new ArrayList<>(this.instrucciones);
            return clonado;
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}

Paso 2: Uso - Código Correcto (✅)

// Crear un prototipo
Envio prototipo = new EnvioBuilder("ENV-PROTO", "Buenos Aires", "Córdoba")
    .descripcion("Medicinas")
    .peso(2.5)
    .fragil(true)
    .requiereSignatura(true)
    .requiereRefrigeracion(true)
    .requiereAseguranza(true)
    .instruccionesEspeciales("Mantener frío")
    .contactoEmergencia("Dr. García")
    .horaEntregaPreferida(LocalTime.of(14, 0))
    .build();

// Crear 100 clones - ¡Mucho más rápido!
for (int i = 0; i < 100; i++) {
    Envio clon = prototipo.clone();
    clon.setId("ENV-" + i);
    envios.add(clon);
}

⚠️ Importante: Shallow Copy vs Deep Copy

Tipo Qué Copia Problema Solución
Shallow Copy Valores primitivos Comparte referencias a objetos Cambiar lista afecta todos los clones
Deep Copy Todo, incluyendo objetos internos Más lento Cada clon es completamente independiente

✅ Ventajas

  • Crear objetos es más rápido que con Builder
  • Útil cuando tienes muchos atributos
  • Útil cuando tienes objetos complejos

📊 Comparación: Cuándo Usar Cada Patrón

Patrón Cuándo Usar Ejemplo Complejidad
Singleton Necesitas exactamente 1 instancia Logger, Configuración Baja
Factory Method Crear 1 tipo de objeto, esconder clase VehiculoFactory.crear("auto") Baja-Media
Abstract Factory Crear FAMILIAS de objetos relacionados VehiculoFactoryArgentina Media
Builder Objetos complejos con muchas opciones Envio con 12+ atributos Media-Alta
Prototype Clonar objetos existentes Crear 100 envios iguales Media

Árbol de Decisión

¿Necesitas crear objetos?
├─ ¿Necesitas exactamente 1 instancia?
│  └─ SÍ → Singleton
├─ ¿Necesitas crear familias de objetos?
│  └─ SÍ → Abstract Factory
├─ ¿El objeto es muy complejo con muchas opciones?
│  └─ SÍ → Builder
├─ ¿Necesitas clonar objetos existentes?
│  └─ SÍ → Prototype
└─ ¿Necesitas esconder la clase concreta?
   └─ SÍ → Factory Method

📋 Resumen: Los 5 Patrones Creacionales

Patrón Problema Solución Complejidad
Singleton 1 instancia única Constructor privado + getInstance() Baja

Abstract Factory Familias de objetos Factory de factories Media
Builder Objetos complejos Construcción paso a paso Media-Alta
Prototype Clonar objetos Implementar Cloneable Media
