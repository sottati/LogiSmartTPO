Clase 9: Patrones Estructurales I

Adapter, Bridge, Composite - Organizando la Complejidad Estructural

Introducción: De la Creación a la Estructura

En las clases anteriores, aprendimos a crear objetos de forma elegante usando patrones creacionales (Singleton, Factory Method, Abstract Factory, Builder, Prototype). Ahora enfrentamos un nuevo desafío: ¿cómo organizamos esos objetos una vez creados?

Los patrones estructurales responden a esta pregunta. Se enfocen en cómo componer objetos y clases en estructuras más grandes, manteniendo flexibilidad y eficiencia. Son sobre relaciones entre objetos, no sobre cómo crearlos.

Los 7 Patrones Estructurales GoF

El Gang of Four identificó 7 patrones estructurales. En esta clase cubriremos los primeros 3:

  • Adapter: Permite que interfaces incompatibles trabajen juntas
  • Bridge: Desacopla una abstracción de su implementación
  • Composite: Compone objetos en estructuras de árbol
  • Decorator: Agrega responsabilidades dinámicamente
  • Facade: Proporciona interfaz simplificada
  • Flyweight: Comparte objetos para economizar memoria
  • Proxy: Proporciona sustituto o marcador de posición

¿Por Qué Estos Patrones Importan?

Imagina que estás integrando múltiples sistemas heredados en tu aplicación. Cada uno tiene su propia interfaz. Imagina que necesitas representar una jerarquía de componentes (como un árbol de carpetas). Imagina que necesitas permitir que diferentes implementaciones de un concepto coexistan sin acoplamiento.

Estos son los problemas que los patrones estructurales resuelven. Son pegamento que mantiene los sistemas complejos unidos sin crear dependencias rígidas.

Analogía: Construcción de un Edificio

Los patrones creacionales son como máquinas que fabrican ladrillos. Los patrones estructurales son como planos que dicen cómo apilar esos ladrillos para construir paredes, pisos y estructuras complejas.

🔌 Adapter Pattern

El Problema: Interfaces Incompatibles

Tienes un sistema que funciona perfectamente. Pero ahora necesitas integrar un componente externo (una librería, un servicio, un sistema heredado) que tiene una interfaz completamente diferente. No puedes cambiar el componente externo, y cambiar tu sistema es costoso.

Ejemplo Real: Adaptador de Corriente

Tu laptop necesita 110V. Viajas a Europa donde el estándar es 220V. ¿Qué haces? Usas un adaptador. El adaptador no cambia ni tu laptop ni el enchufe europeo. Simplemente traduce entre los dos estándares.

La Solución: El Patrón Adapter

El patrón Adapter actúa como un traductor entre dos interfaces incompatibles. Crea una clase intermedia que implementa la interfaz que tu sistema espera, pero internamente delega al componente externo, traduciendo las llamadas según sea necesario.

Estructura del Patrón Adapter

Dos Variantes: Class Adapter vs Object Adapter

1. Class Adapter (Herencia)

El Adapter hereda tanto de la interfaz Target como del Adaptee. Menos común en lenguajes modernos.

// Class Adapter - Herencia múltiple public class Adapter extends Adaptee implements Target { @Override public void operacionEsperada() { // Traduce a la operación del Adaptee this.operacionDelAdaptee(); } }

2. Object Adapter (Composición)

El Adapter contiene una instancia del Adaptee y delega a través de composición. Es más flexible y es la variante preferida.

// Object Adapter - Composición public class Adapter implements Target { private Adaptee adaptee; public Adapter(Adaptee adaptee) { this.adaptee = adaptee; } @Override public void operacionEsperada() { // Traduce la llamada adaptee.operacionDelAdaptee(); } }

Ejemplo Completo: Integrando un Sistema de Pagos Heredado

Situación: Tu sistema de e-commerce usa una interfaz de pagos moderna. Pero necesitas soportar un sistema de pagos heredado que tiene una interfaz completamente diferente.

// Interfaz que tu sistema espera (Target) public interface ProcesadorPagos { boolean procesarPago(double monto, String tarjeta); String obtenerEstado(); } // Sistema heredado (Adaptee) - No puedes cambiar esto public class SistemaPagosLegacy { public int procesarTransaccion(float cantidad, String numeroTarjeta) { // Lógica heredada return 1; // 1 = éxito, 0 = fallo } public String consultarEstado() { return "COMPLETADO"; } } // El Adapter - Traduce entre los dos public class AdapterPagosLegacy implements ProcesadorPagos { private SistemaPagosLegacy sistemaPagosLegacy; public AdapterPagosLegacy(SistemaPagosLegacy sistemaPagosLegacy) { this.sistemaPagosLegacy = sistemaPagosLegacy; } @Override public boolean procesarPago(double monto, String tarjeta) { // Traduce: double -> float, boolean -> int int resultado = sistemaPagosLegacy.procesarTransaccion((float)monto, tarjeta); return resultado == 1; } @Override public String obtenerEstado() { return sistemaPagosLegacy.consultarEstado(); } } // Uso SistemaPagosLegacy sistemaPagosLegacy = new SistemaPagosLegacy(); ProcesadorPagos procesador = new AdapterPagosLegacy(sistemaPagosLegacy); // Ahora tu sistema puede usar el procesador como si fuera moderno boolean exito = procesador.procesarPago(99.99, "4111-1111-1111-1111"); String estado = procesador.obtenerEstado();

Ventajas y Desventajas

Ventajas Desventajas
✅ Permite integrar componentes incompatibles ❌ Agrega una capa de indirección
✅ No requiere cambiar el código existente ❌ Puede ser excesivo si solo adaptas un método
✅ Sigue el Principio de Abierto/Cerrado ❌ A veces es difícil adaptar interfaces muy diferentes
✅ Facilita testing (puedes mockear el adaptee)

¿Cuándo Usar Adapter?

  • Necesitas integrar una librería externa con interfaz incompatible
  • Trabajas con código heredado que no puedes cambiar
  • Quieres permitir múltiples implementaciones de una interfaz
  • Necesitas traducir entre diferentes convenciones (ej: métodos, tipos de datos)
Duración de esta sección: ~20 minutos

🌉 Bridge Pattern

El Problema: Explosión de Subclases

Tienes una clase base `Vehículo`. Necesitas soportar múltiples tipos: `Auto`, `Moto`, `Camión`. Pero también necesitas múltiples implementaciones de cada tipo según el motor: `Gasolina`, `Diésel`, `Eléctrico`.

Si usas herencia pura, terminas con una explosión combinatoria: `AutoGasolina`, `AutoDiésel`, `AutoEléctrico`, `MotoGasolina`, `MotoDiésel`, `MotoEléctrico`, etc. ¡9 clases para 3 tipos × 3 motores!

El Problema Visualizado

Herencia Pura (Problema): Vehículo / | \ Auto Moto Camión / | \ / | \ / | \ G D E G D E G D E (G=Gasolina, D=Diésel, E=Eléctrico) Total: 9 clases, difícil de mantener

La Solución: Separar Abstracción de Implementación

El patrón Bridge resuelve esto separando dos dimensiones de variación en dos jerarquías independientes:

  • Abstracción: La interfaz de alto nivel (Vehículo)
  • Implementación: Los detalles específicos (Motor)

Estructura del Patrón Bridge

┌──────────────────────────────────────────────┐ │ Abstracción (Vehículo) │ │ - Tipo: Auto, Moto, Camión │ └──────────────┬───────────────────────────────┘ │ │ usa (Bridge) ↓ ┌──────────────────────────────────────────────┐ │ Implementación (Motor) │ │ - Gasolina, Diésel, Eléctrico │ └──────────────────────────────────────────────┘ Resultado: 3 + 3 = 6 clases (no 9)

Ejemplo Completo: Sistema de Vehículos

// Implementación (Motor) - Primera jerarquía public interface Motor { void arrancar(); void acelerar(); void detener(); } public class MotorGasolina implements Motor { @Override public void arrancar() { System.out.println("Motor de gasolina arrancando..."); } @Override public void acelerar() { System.out.println("Acelerando con gasolina"); } @Override public void detener() { System.out.println("Motor de gasolina detenido"); } } public class MotorDiesel implements Motor { @Override public void arrancar() { System.out.println("Motor diésel arrancando..."); } @Override public void acelerar() { System.out.println("Acelerando con diésel"); } @Override public void detener() { System.out.println("Motor diésel detenido"); } } // Abstracción (Vehículo) - Segunda jerarquía public abstract class Vehiculo { protected Motor motor; // El Bridge public Vehiculo(Motor motor) { this.motor = motor; } public abstract void conducir(); } public class Auto extends Vehiculo { public Auto(Motor motor) { super(motor); } @Override public void conducir() { motor.arrancar(); motor.acelerar(); System.out.println("Auto conduciendo suavemente"); motor.detener(); } } public class Moto extends Vehiculo { public Moto(Motor motor) { super(motor); } @Override public void conducir() { motor.arrancar(); motor.acelerar(); System.out.println("Moto acelerando rápidamente"); motor.detener(); } } // Uso Motor motorGasolina = new MotorGasolina(); Motor motorDiesel = new MotorDiesel(); Vehiculo auto = new Auto(motorGasolina); Vehiculo moto = new Moto(motorDiesel); auto.conducir(); // Auto con gasolina moto.conducir(); // Moto con diésel // Puedo cambiar el motor sin cambiar la clase del vehículo Vehiculo autoConDiesel = new Auto(motorDiesel); autoConDiesel.conducir();

Bridge vs Adapter: ¿Cuál es la Diferencia?

Aspecto Adapter Bridge
Propósito Hacer que interfaces incompatibles funcionen juntas Separar abstracción de implementación
Cuándo se aplica Después (integración de componentes existentes) Antes (diseño de nuevos sistemas)
Estructura Traductor entre dos interfaces Separación de dos dimensiones de variación
Flexibilidad Permite trabajar con lo que ya existe Permite evolucionar independientemente

Ventajas y Desventajas

Ventajas Desventajas
✅ Evita explosión de subclases ❌ Más complejo que herencia simple
✅ Permite variar abstracción e implementación independientemente ❌ Requiere identificar las dimensiones de variación
✅ Facilita agregar nuevas abstracciones o implementaciones ❌ Puede ser excesivo para sistemas simples
✅ Sigue el Principio de Responsabilidad Única

¿Cuándo Usar Bridge?

  • Tienes dos dimensiones de variación independientes
  • Quieres evitar explosión de subclases
  • Necesitas que abstracción e implementación evolucionen independientemente
  • Quieres compartir implementaciones entre múltiples abstracciones
Duración de esta sección: ~25 minutos

🎨 Composite Pattern

El Problema: Jerarquías de Componentes

Necesitas representar una estructura jerárquica: un árbol de carpetas con archivos, un menú con submenús, un organigrama de empleados. El desafío es que necesitas tratar tanto los componentes individuales como los contenedores de forma uniforme.

Ejemplo: Sistema de Archivos

Una carpeta puede contener archivos y otras carpetas. Un archivo no puede contener nada. Pero cuando calculas el tamaño total de una carpeta, necesitas sumar recursivamente todos los archivos dentro, incluyendo los en subcarpetas.

La Solución: Estructura Recursiva Uniforme

El patrón Composite permite componer objetos en estructuras de árbol para representar jerarquías parte-todo. Los clientes pueden tratar objetos individuales y composiciones de objetos de forma uniforme.

Estructura del Patrón Composite

Component (Interfaz) / \ / \ Leaf Composite (Archivo) (Carpeta) / \ / \ children[] operaciones (add, remove)

Ejemplo Completo: Sistema de Archivos

// Componente base public abstract class ComponenteArchivo { protected String nombre; public ComponenteArchivo(String nombre) { this.nombre = nombre; } public abstract int obtenerTamano(); public abstract void mostrar(int profundidad); } // Hoja: Archivo public class Archivo extends ComponenteArchivo { private int tamano; public Archivo(String nombre, int tamano) { super(nombre); this.tamano = tamano; } @Override public int obtenerTamano() { return tamano; } @Override public void mostrar(int profundidad) { System.out.println(" ".repeat(profundidad) + "📄 " + nombre + " (" + tamano + " KB)"); } } // Composite: Carpeta public class Carpeta extends ComponenteArchivo { private List componentes = new ArrayList<>(); public Carpeta(String nombre) { super(nombre); } public void agregar(ComponenteArchivo componente) { componentes.add(componente); } public void remover(ComponenteArchivo componente) { componentes.remove(componente); } @Override public int obtenerTamano() { int totalTamano = 0; for (ComponenteArchivo componente : componentes) { totalTamano += componente.obtenerTamano(); } return totalTamano; } @Override public void mostrar(int profundidad) { System.out.println(" ".repeat(profundidad) + "📁 " + nombre); for (ComponenteArchivo componente : componentes) { componente.mostrar(profundidad + 1); } } } // Uso Carpeta raiz = new Carpeta("Documentos"); Carpeta proyectos = new Carpeta("Proyectos"); Carpeta trabajo = new Carpeta("Trabajo"); Archivo archivo1 = new Archivo("Reporte.pdf", 500); Archivo archivo2 = new Archivo("Presupuesto.xlsx", 300); Archivo archivo3 = new Archivo("Contrato.docx", 200); proyectos.agregar(archivo1); proyectos.agregar(archivo2); trabajo.agregar(archivo3); raiz.agregar(proyectos); raiz.agregar(trabajo); raiz.mostrar(0); System.out.println("Tamaño total: " + raiz.obtenerTamano() + " KB"); // Salida: // 📁 Documentos // 📁 Proyectos // 📄 Reporte.pdf (500 KB) // 📄 Presupuesto.xlsx (300 KB) // 📁 Trabajo // 📄 Contrato.docx (200 KB) // Tamaño total: 1000 KB

Variantes: Seguridad vs Transparencia

Enfoque Transparente (Mostrado Arriba)

Tanto Leaf como Composite implementan todas las operaciones. Los Leaf simplemente no hacen nada (o lanzan excepciones) para operaciones de contenedor.

Enfoque Seguro

Solo Composite implementa operaciones de contenedor. Los clientes deben verificar el tipo antes de usar operaciones específicas.

// Enfoque Seguro public abstract class ComponenteArchivo { public abstract int obtenerTamano(); public abstract void mostrar(int profundidad); // Solo en Composite public void agregar(ComponenteArchivo c) { throw new UnsupportedOperationException(); } } // Uso con verificación de tipo if (componente instanceof Carpeta) { ((Carpeta) componente).agregar(nuevoComponente); }

Ventajas y Desventajas

Ventajas Desventajas
✅ Simplifica código cliente (trata hojas y composites igual) ❌ Puede hacer el diseño demasiado general
✅ Fácil agregar nuevos tipos de componentes ❌ Difícil restringir tipos de componentes en un contenedor
✅ Representa jerarquías naturalmente ❌ Puede llevar a un árbol profundo y complejo
✅ Sigue el Principio de Abierto/Cerrado

¿Cuándo Usar Composite?

  • Necesitas representar jerarquías parte-todo
  • Quieres que los clientes ignoren la diferencia entre objetos y composiciones
  • La estructura es recursiva por naturaleza
  • Necesitas operaciones que funcionen en toda la jerarquía
Duración de esta sección: ~20 minutos

Decisión: ¿Cuál Usar?

Pregúntate:

  • ¿Necesito integrar código externo incompatible? → Usa Adapter
  • ¿Tengo dos dimensiones de variación que necesito separar? → Usa Bridge
  • ¿Necesito representar una jerarquía parte-todo? → Usa Composite

🎯 Diferencia clave rápida

Patrón ¿Qué resuelve? Cuándo elegir
Adapter Interfaces incompatibles "Necesito usar esta librería pero su API no coincide"
Bridge Dos dimensiones de variación "Voy a tener muchos tipos de X combinados con muchos tipos de Y"
Composite Objetos individuales vs grupos "Tengo carpetas que contienen archivos y más carpetas"

🔗 Relaciones entre patrones

  • Bridge ↔ Adapter: ambos usan composición, pero Bridge previene problemas mientras Adapter corrige
  • Bridge ↔ Composite: Bridge se relaciona con Composite por uso de composición
  • Composite ↔ Decorator: ambos recursivos, pero:
    • Composite: construye jerarquías de árbol
    • Decorator: agrega responsabilidades dinámicamente

🏢 Casos de Aplicación en Sistemas Reales

1. Adapter en el Mundo Real

Caso: Integración de Múltiples Proveedores de Email

Tu aplicación necesita enviar emails. Inicialmente usas Gmail. Luego necesitas soportar Outlook, SendGrid y un servidor SMTP personalizado. Cada uno tiene su propia API.

// Interfaz que tu aplicación espera public interface ServicioEmail { void enviar(String destinatario, String asunto, String cuerpo); } // Adaptadores para cada proveedor public class AdapterGmail implements ServicioEmail { private GmailAPI gmailAPI; @Override public void enviar(String destinatario, String asunto, String cuerpo) { gmailAPI.sendMessage(destinatario, asunto, cuerpo); } } public class AdapterSendGrid implements ServicioEmail { private SendGridClient sendGrid; @Override public void enviar(String destinatario, String asunto, String cuerpo) { sendGrid.send(new Email(destinatario, asunto, cuerpo)); } } // Tu aplicación usa cualquiera sin cambios ServicioEmail email = new AdapterGmail(); // o AdapterSendGrid email.enviar("usuario@example.com", "Bienvenido", "Hola!");

2. Bridge en el Mundo Real

Caso: Sistema de Notificaciones Multi-Canal

Necesitas enviar notificaciones por email, SMS, push y Slack. Cada canal tiene múltiples proveedores (Gmail, SendGrid para email; Twilio, AWS SNS para SMS, etc.).

// Abstracción: Tipo de notificación public abstract class Notificacion { protected ProveedorNotificacion proveedor; public Notificacion(ProveedorNotificacion proveedor) { this.proveedor = proveedor; } public abstract void enviar(String usuario, String mensaje); } // Implementación: Proveedor específico public interface ProveedorNotificacion { void enviarMensaje(String destino, String contenido); } // Concretos public class NotificacionUrgente extends Notificacion { @Override public void enviar(String usuario, String mensaje) { proveedor.enviarMensaje(usuario, "[URGENTE] " + mensaje); } } public class NotificacionMarketing extends Notificacion { @Override public void enviar(String usuario, String mensaje) { proveedor.enviarMensaje(usuario, "[MARKETING] " + mensaje); } } // Proveedores public class ProveedorEmail implements ProveedorNotificacion { @Override public void enviarMensaje(String destino, String contenido) { // Envía por email } } public class ProveedorSMS implements ProveedorNotificacion { @Override public void enviarMensaje(String destino, String contenido) { // Envía por SMS } } // Uso ProveedorNotificacion email = new ProveedorEmail(); ProveedorNotificacion sms = new ProveedorSMS(); Notificacion urgente = new NotificacionUrgente(email); Notificacion marketing = new NotificacionMarketing(sms); urgente.enviar("juan@example.com", "Tu pago fue procesado"); marketing.enviar("+5491234567890", "¡Descuento especial!");

3. Composite en el Mundo Real

Caso: Sistema de Permisos Jerárquico

Un sistema de gestión de permisos donde los permisos pueden ser individuales o grupos de permisos. Un usuario tiene ciertos permisos directos, pero también hereda permisos de sus grupos.

// Componente base public abstract class Permiso { public abstract boolean tienePermiso(String accion); } // Hoja: Permiso individual public class PermisoIndividual extends Permiso { private String accion; public PermisoIndividual(String accion) { this.accion = accion; } @Override public boolean tienePermiso(String accion) { return this.accion.equals(accion); } } // Composite: Grupo de permisos public class GrupoPermisos extends Permiso { private List permisos = new ArrayList<>(); public void agregar(Permiso permiso) { permisos.add(permiso); } @Override public boolean tienePermiso(String accion) { for (Permiso p : permisos) { if (p.tienePermiso(accion)) { return true; } } return false; } } // Uso GrupoPermisos grupoAdmin = new GrupoPermisos(); grupoAdmin.agregar(new PermisoIndividual("crear_usuario")); grupoAdmin.agregar(new PermisoIndividual("eliminar_usuario")); grupoAdmin.agregar(new PermisoIndividual("editar_configuracion")); GrupoPermisos grupoModerador = new GrupoPermisos(); grupoModerador.agregar(new PermisoIndividual("moderar_comentarios")); grupoModerador.agregar(new PermisoIndividual("bloquear_usuario")); // Usuario hereda permisos de ambos grupos GrupoPermisos permisoUsuario = new GrupoPermisos(); permisoUsuario.agregar(grupoAdmin); permisoUsuario.agregar(grupoModerador); permisoUsuario.agregar(new PermisoIndividual("cambiar_contraseña")); // Verificar permisos System.out.println(permisoUsuario.tienePermiso("crear_usuario")); // true System.out.println(permisoUsuario.tienePermiso("moderar_comentarios")); // true System.out.println(permisoUsuario.tienePermiso("cambiar_contraseña")); // true System.out.println(permisoUsuario.tienePermiso("acceso_root")); // false
Duración de esta sección: ~10 minutos

🎯 Hito 8: Actividad Práctica - Data Flow con Patrones Estructurales

Contexto: Continuando LogiSmart

En los hitos anteriores, construiste el modelo de dominio de LogiSmart, lo implementaste en Java, y aplicaste patrones GRASP y GoF creacionales. Ahora, en el Hito 6, vamos a integrar nuevas fuentes de datos y componentes externos usando patrones estructurales.

Situación de Negocio

LogiSmart ha crecido. Ahora necesita:

  • Integrar datos de múltiples proveedores de logística (DHL, FedEx, UPS) que tienen APIs diferentes
  • Soportar múltiples tipos de reportes (PDF, Excel, JSON) con diferentes motores de generación
  • Representar una estructura jerárquica de centros de distribución con almacenes y puntos de entrega

Objetivos del Hito 8

Debes aplicar los tres patrones estructurales:

  • Adapter: Integrar APIs de proveedores externos (DHL, FedEx, UPS)
  • Bridge: Separar tipos de reportes de motores de generación
  • Composite: Modelar la jerarquía de centros de distribución
