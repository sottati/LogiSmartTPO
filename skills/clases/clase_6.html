¿Qué son los Patrones de Diseño?

Definición

Un Patrón de Diseño es una solución reutilizable a un problema común en el diseño de software. Es una descripción de cómo resolver un problema, no el código en sí.

Analogía: Un arquitecto no inventa cómo construir una puerta cada vez que diseña una casa. Usa patrones probados. Lo mismo hacemos en software.

Características de un Buen Patrón

Característica Descripción
Resuelve un problema específico No es una solución genérica, sino para un contexto particular
Es probado Otros lo han usado exitosamente
Es documentado Tiene un nombre, descripción, ventajas y desventajas
Es reutilizable Puedes aplicarlo en diferentes contextos

Estructura de un Patrón

Ejemplo: Patrón Singleton

Nombre: Singleton

Problema: Necesito asegurar que una clase tenga una única instancia

Solución: Hacer el constructor privado y proporcionar un método estático para obtener la instancia

Ventajas: Control global, acceso único

Desventajas: Difícil de testear, oculta dependencias

Historia: Gang of Four (GoF)

En 1994, cuatro autores publicaron el libro "Design Patterns: Elements of Reusable Object-Oriented Software". Estos autores son conocidos como el Gang of Four (GoF):

  • Erich Gamma
  • Richard Helm
  • Ralph Johnson
  • John Vlissides

El libro documentó 23 patrones que se convirtieron en el estándar de la industria. Desde entonces, se han descubierto más patrones, pero los 23 del GoF siguen siendo los más importantes.

Clasificación GoF

Los 23 patrones del GoF se clasifican en tres categorías según el problema que resuelven:

1. Patrones Creacionales (5 patrones)

Resuelven problemas de cómo crear objetos. El objetivo es crear objetos de forma flexible, sin acoplar el código al tipo específico de objeto.

Patrón Descripción
Singleton Una única instancia en toda la aplicación
Factory Method Crear objetos sin especificar su clase exacta
Abstract Factory Crear familias de objetos relacionados
Builder Construir objetos complejos paso a paso
Prototype Crear objetos clonando un prototipo

2. Patrones Estructurales (7 patrones)

Resuelven problemas de cómo organizar las relaciones entre objetos. El objetivo es crear estructuras flexibles que combinen objetos y clases.

Patrón Descripción
Adapter Hacer compatible una interfaz con otra
Bridge Desacoplar abstracción de implementación
Composite Tratar objetos individuales y composiciones uniformemente
Decorator Agregar responsabilidades dinámicamente
Facade Proporcionar interfaz simplificada a un subsistema
Flyweight Compartir objetos para ahorrar memoria
Proxy Proporcionar un substituto o placeholder

3. Patrones de Comportamiento (11 patrones)

Resuelven problemas de cómo comunicarse entre objetos. El objetivo es definir cómo los objetos interactúan y distribuyen responsabilidades.

Patrón Descripción
Chain of Responsibility Pasar una solicitud a lo largo de una cadena de objetos
Command Encapsular una solicitud como un objeto
Interpreter Definir una representación de un lenguaje
Iterator Acceder secuencialmente a elementos de una colección
Mediator Definir un objeto que encapsula cómo interactúan otros objetos
Memento Capturar y externalizar el estado de un objeto
Observer Notificar a múltiples objetos sobre cambios de estado
State Permitir que un objeto altere su comportamiento según su estado
Strategy Encapsular una familia de algoritmos intercambiables
Template Method Definir el esqueleto de un algoritmo en una clase base
Visitor Representar una operación a realizar sobre elementos de una estructura

Resumen

Categoría Cantidad Objetivo Pregunta
Creacionales 5 Crear objetos ¿Cómo creo esto?
Estructurales 7 Organizar relaciones ¿Cómo organizo esto?
Comportamiento 11 Comunicación ¿Cómo se comunican?

Patrones Creacionales: Conceptos

¿Por qué son importantes?

Imagina que tienes un sistema donde necesitas crear muchos tipos de objetos: Envios, Vehiculos, Notificadores, etc. Si el código cliente sabe exactamente qué clase instanciar, entonces:

  1. El código está fuertemente acoplado a las clases específicas
  2. Si necesitas cambiar cómo se crean los objetos, tienes que cambiar el código cliente
  3. Es difícil testear porque siempre creas objetos reales

Principio Fundamental

"Depende de abstracciones, no de concreciones."

Comparación: Antes vs Después

❌ Sin Patrón

Envio envio = new EnvioExpress(); // Acoplado a EnvioExpress // Si cambias la clase, cambias aquí

✅ Con Patrón

Envio envio = factory.crearEnvio( TipoEnvio.EXPRESS ); // Desacoplado // Cambios centralizados

Los 5 Patrones Creacionales

Patrón Cuándo Usar Complejidad
Singleton Una única instancia (logger, config, BD) Baja
Factory Method Múltiples subclases, crear la correcta Media
Abstract Factory Familias de objetos relacionados Alta
Builder Objetos complejos con muchos parámetros Media
Prototype Crear objetos es costoso, clonar es barato Media

Ventajas Generales

  • Flexibilidad: Fácil cambiar qué objetos se crean
  • Mantenibilidad: Cambios centralizados en la creación
  • Testabilidad: Fácil inyectar mocks
  • Escalabilidad: Fácil agregar nuevos tipos de objetos

Singleton: Una Única Instancia

El Problema

Imagina que necesitas una clase que gestione la configuración del sistema. Necesitas que:

  1. Haya una única instancia en toda la aplicación
  2. Todos accedan a la misma instancia
  3. No se pueda crear otra instancia

Sin Singleton (❌ Incorrecto)

Config config1 = new Config(); Config config2 = new Config(); // ❌ Ahora tienes dos instancias diferentes // Si cambias config1, config2 no se entera

Con Singleton (✅ Correcto)

class Config { private static Config instancia; private Config() { // Constructor privado // Inicialización } public static Config obtenerInstancia() { if (instancia == null) { instancia = new Config(); } return instancia; } } // Uso: Config config = Config.obtenerInstancia();

Problema: Thread Safety

En un sistema multihilo, el código anterior tiene un problema. Dos hilos podrían crear dos instancias simultáneamente.

Soluciones

Solución 1: Sincronizar

public static synchronized Config obtenerInstancia() { if (instancia == null) { instancia = new Config(); } return instancia; }

Solución 2: Eager Initialization

class Config { private static final Config instancia = new Config(); private Config() {} public static Config obtenerInstancia() { return instancia; } }

Solución 3: Double-Checked Locking

public static Config obtenerInstancia() { if (instancia == null) { synchronized (Config.class) { if (instancia == null) { instancia = new Config(); } } } return instancia; }

Ventajas y Desventajas

Ventajas Desventajas
Único punto de acceso Difícil de testear
Lazy initialization Oculta dependencias
Acceso global Puede ser un anti-patrón

Cuándo Usar

  • Conexión a base de datos
  • Logger del sistema
  • Configuración de la aplicación
  • Pool de conexiones
  • Cache global

Factory Method: Crear Sin Acoplar

El Problema

En LogiSmart tienes diferentes tipos de envíos:

  • EnvioExpress (24 horas)
  • EnvioStandard (3-5 días)
  • EnvioEconomico (7-10 días)

Sin Factory Method (❌ Acoplamiento)

String tipo = "express"; Envio envio; if (tipo.equals("express")) { envio = new EnvioExpress(); } else if (tipo.equals("standard")) { envio = new EnvioStandard(); } else { envio = new EnvioEconomico(); } // ❌ Código cliente acoplado a todas las subclases // ❌ Si agregas un tipo, cambias aquí

Con Factory Method (✅ Desacoplado)

interface Envio { void enviar(); } class EnvioExpress implements Envio { public void enviar() { /* ... */ } } class FabricaDeEnvios { public static Envio crearEnvio(String tipo) { switch (tipo) { case "express": return new EnvioExpress(); case "standard": return new EnvioStandard(); default: return new EnvioEconomico(); } } } // Uso: Envio envio = FabricaDeEnvios.crearEnvio("express"); // ✅ Código cliente desacoplado // ✅ Cambios centralizados en la factory

Ventajas de Factory Method

  • Desacoplamiento: El código cliente no conoce las subclases
  • Centralización: La creación está en un solo lugar
  • Flexibilidad: Fácil agregar nuevos tipos
  • Testabilidad: Fácil inyectar mocks

Cuándo Usar

  • Tienes múltiples subclases y quieres crear la correcta dinámicamente
  • La creación es compleja
  • Quieres desacoplar el código cliente de las subclases

En LogiSmart: 3 Ejemplos

1. Factory de Envíos

class FabricaDeEnvios { public static Envio crearEnvio(TipoEnvio tipo, Cliente cliente, Ubicacion origen, Ubicacion destino) { switch (tipo) { case EXPRESS: return new EnvioExpress(cliente, origen, destino); case STANDARD: return new EnvioStandard(cliente, origen, destino); case ECONOMICO: return new EnvioEconomico(cliente, origen, destino); default: throw new IllegalArgumentException("Tipo inválido"); } } }

2. Factory de Notificadores

class FabricaDeNotificadores { public static Notificador crearNotificador(TipoNotificador tipo) { switch (tipo) { case EMAIL: return new NotificadorEmail(); case SMS: return new NotificadorSMS(); case PUSH: return new NotificadorPush(); default: throw new IllegalArgumentException("Tipo inválido"); } } }

3. Factory de Vehículos

class FabricaDeVehiculos { public static Vehiculo crearVehiculo(TipoVehiculo tipo) { switch (tipo) { case AUTO: return new Auto(); case CAMION: return new Camion(); case MOTO: return new Moto(); default: throw new IllegalArgumentException("Tipo inválido"); } } }

Comparación: Singleton vs Factory Method

Aspecto Singleton Factory Method
Objetivo Una única instancia Crear objetos del tipo correcto
Uso Acceso global Creación flexible
Acoplamiento Alto (depende de la clase) Bajo (depende de interfaz)
Testabilidad Difícil Fácil
Complejidad Simple Moderada
Ejemplo Base de datos Tipos de envíos

¿Cuándo Usar Cada Uno?

Usa Singleton Cuando:

  • Necesitas un único punto de acceso global
  • Quieres controlar la instancia única
  • La creación es costosa y quieres lazy initialization

Usa Factory Method Cuando:

  • Tienes múltiples tipos de objetos
  • Quieres desacoplar el código cliente
  • La creación es compleja o varía según el contexto
