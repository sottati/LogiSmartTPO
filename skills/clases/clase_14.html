🎯 Introducción

Objetivo de la clase: Entender patrones de acceso a datos y cómo diseñar una arquitectura lógica que separe la lógica de negocio de la persistencia.

¿Por qué patrones de acceso a datos?

Los patrones de acceso a datos resuelven un problema fundamental en aplicaciones empresariales: cómo interactúan los objetos en memoria con datos persistentes en bases de datos.

Problema: Sin estos patrones, el código de negocio se mezcla con SQL, lo que causa:
  • Código difícil de probar (tests requieren BD real)
  • Cambios en BD afectan toda la aplicación
  • Imposible cambiar de BD sin reescribir código
  • Duplicación de lógica de persistencia

Patrones que cubriremos

Patrón Propósito Cuándo usar
Data Mapper Mapear objetos a tablas Cuando necesitas independencia entre modelo de objetos y BD
Repository Acceso uniforme a datos Cuando necesitas abstraer la fuente de datos
Unit of Work Gestionar transacciones Cuando necesitas coordinar múltiples cambios
Lazy Load Cargar datos bajo demanda Cuando necesitas optimizar rendimiento

🗺️ Data Mapper

Data Mapper es un patrón que mapea objetos de dominio a registros de base de datos de forma independiente.

Problema

¿Cómo mantener objetos de dominio independientes de la estructura de la BD?

┌─────────────────────────────────────────────────┐ │ SIN Data Mapper (Acoplamiento) │ ├─────────────────────────────────────────────────┤ │ Objeto Envio │ │ ├─ id │ │ ├─ origen │ │ ├─ destino │ │ ├─ peso │ │ └─ save() → SQL directo │ │ │ │ Problema: Cambio en BD = Cambio en objeto │ └─────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────┐ │ CON Data Mapper (Desacoplamiento) │ ├─────────────────────────────────────────────────┤ │ Objeto Envio (Puro) │ │ ├─ id │ │ ├─ origen │ │ ├─ destino │ │ └─ peso │ │ │ │ EnvioMapper │ │ ├─ insert(envio) → SQL │ │ ├─ update(envio) → SQL │ │ ├─ delete(envio) → SQL │ │ └─ findById(id) → Envio │ │ │ │ Ventaja: Objeto puro, BD independiente │ └─────────────────────────────────────────────────┘

Solución

Crear un Mapper que traduce entre objetos de dominio y registros de BD.

Paso 1: Objeto de Dominio (Puro)

public class Envio {
    private int id;
    private String origen;
    private String destino;
    private double peso;
    
    public Envio(int id, String origen, String destino, double peso) {
        this.id = id;
        this.origen = origen;
        this.destino = destino;
        this.peso = peso;
    }
    
    // Getters
    public int getId() { return id; }
    public String getOrigen() { return origen; }
    public String getDestino() { return destino; }
    public double getPeso() { return peso; }
}

Paso 2: Data Mapper

public class EnvioMapper {
    private Connection conexion;
    
    public EnvioMapper(Connection conexion) {
        this.conexion = conexion;
    }
    
    // Insertar
    public void insert(Envio envio) {
        String sql = "INSERT INTO envios (id, origen, destino, peso) VALUES (?, ?, ?, ?)";
        try (PreparedStatement stmt = conexion.prepareStatement(sql)) {
            stmt.setInt(1, envio.getId());
            stmt.setString(2, envio.getOrigen());
            stmt.setString(3, envio.getDestino());
            stmt.setDouble(4, envio.getPeso());
            stmt.executeUpdate();
            System.out.println("✓ Envío insertado");
        } catch (SQLException e) {
            System.out.println("✗ Error al insertar: " + e.getMessage());
        }
    }
    
    // Actualizar
    public void update(Envio envio) {
        String sql = "UPDATE envios SET origen=?, destino=?, peso=? WHERE id=?";
        try (PreparedStatement stmt = conexion.prepareStatement(sql)) {
            stmt.setString(1, envio.getOrigen());
            stmt.setString(2, envio.getDestino());
            stmt.setDouble(3, envio.getPeso());
            stmt.setInt(4, envio.getId());
            stmt.executeUpdate();
            System.out.println("✓ Envío actualizado");
        } catch (SQLException e) {
            System.out.println("✗ Error al actualizar: " + e.getMessage());
        }
    }
    
    // Eliminar
    public void delete(Envio envio) {
        String sql = "DELETE FROM envios WHERE id=?";
        try (PreparedStatement stmt = conexion.prepareStatement(sql)) {
            stmt.setInt(1, envio.getId());
            stmt.executeUpdate();
            System.out.println("✓ Envío eliminado");
        } catch (SQLException e) {
            System.out.println("✗ Error al eliminar: " + e.getMessage());
        }
    }
    
    // Buscar por ID
    public Envio findById(int id) {
        String sql = "SELECT * FROM envios WHERE id=?";
        try (PreparedStatement stmt = conexion.prepareStatement(sql)) {
            stmt.setInt(1, id);
            ResultSet rs = stmt.executeQuery();
            if (rs.next()) {
                return new Envio(
                    rs.getInt("id"),
                    rs.getString("origen"),
                    rs.getString("destino"),
                    rs.getDouble("peso")
                );
            }
        } catch (SQLException e) {
            System.out.println("✗ Error al buscar: " + e.getMessage());
        }
        return null;
    }
}

Ventajas

  • ✅ Objeto de dominio puro (sin SQL)
  • ✅ Fácil de testear (mock del mapper)
  • ✅ Cambios en BD no afectan objeto
  • ✅ Reutilizable para diferentes BD

Desventajas

  • ❌ Más código (mapper adicional)
  • ❌ Overhead de traducción
  • ❌ Duplicación de propiedades

📦 Repository

Repository es un patrón que proporciona una interfaz uniforme para acceder a datos, ocultando los detalles de persistencia.

Problema

¿Cómo proporcionar una interfaz consistente para acceder a datos sin exponer detalles de persistencia?

┌─────────────────────────────────────────────────┐ │ SIN Repository (Acoplamiento a BD) │ ├─────────────────────────────────────────────────┤ │ Lógica de Negocio │ │ ├─ new EnvioMapper() │ │ ├─ mapper.findById(1) │ │ ├─ mapper.insert(envio) │ │ └─ mapper.update(envio) │ │ │ │ Problema: Lógica acoplada a mapper │ └─────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────┐ │ CON Repository (Desacoplamiento) │ ├─────────────────────────────────────────────────┤ │ Lógica de Negocio │ │ ├─ repository.obtener(1) │ │ ├─ repository.guardar(envio) │ │ └─ repository.eliminar(envio) │ │ │ │ EnvioRepository (Interfaz) │ │ ├─ obtener(id) │ │ ├─ guardar(envio) │ │ ├─ eliminar(envio) │ │ └─ obtenerTodos() │ │ │ │ EnvioRepositorySQL (Implementación) │ │ └─ Usa EnvioMapper internamente │ │ │ │ Ventaja: Lógica agnóstica a persistencia │ └─────────────────────────────────────────────────┘

Solución

Crear una interfaz Repository que define operaciones genéricas.

Paso 1: Interfaz Repository

public interface RepositorioEnvio {
    void guardar(Envio envio);
    void actualizar(Envio envio);
    void eliminar(Envio envio);
    Envio obtener(int id);
    List obtenerTodos();
    List buscar(String criterio);
}

Paso 2: Implementación SQL

public class RepositorioEnvioSQL implements RepositorioEnvio {
    private EnvioMapper mapper;
    
    public RepositorioEnvioSQL(Connection conexion) {
        this.mapper = new EnvioMapper(conexion);
    }
    
    @Override
    public void guardar(Envio envio) {
        mapper.insert(envio);
    }
    
    @Override
    public void actualizar(Envio envio) {
        mapper.update(envio);
    }
    
    @Override
    public void eliminar(Envio envio) {
        mapper.delete(envio);
    }
    
    @Override
    public Envio obtener(int id) {
        return mapper.findById(id);
    }
    
    @Override
    public List obtenerTodos() {
        // Implementación
        return new ArrayList<>();
    }
    
    @Override
    public List buscar(String criterio) {
        // Implementación
        return new ArrayList<>();
    }
}

Paso 3: Implementación en Memoria (para tests)

public class RepositorioEnvioMemoria implements RepositorioEnvio {
    private Map almacen = new HashMap<>();
    
    @Override
    public void guardar(Envio envio) {
        almacen.put(envio.getId(), envio);
        System.out.println("✓ Envío guardado en memoria");
    }
    
    @Override
    public Envio obtener(int id) {
        return almacen.get(id);
    }
    
    @Override
    public List obtenerTodos() {
        return new ArrayList<>(almacen.values());
    }
    
    // Otros métodos...
}

Paso 4: Uso en Lógica de Negocio

public class ServicioEnvios {
    private RepositorioEnvio repositorio;
    
    public ServicioEnvios(RepositorioEnvio repositorio) {
        this.repositorio = repositorio;
    }
    
    public void procesarEnvio(Envio envio) {
        // Lógica de negocio
        repositorio.guardar(envio);
        System.out.println("✓ Envío procesado");
    }
    
    public Envio obtenerEnvio(int id) {
        return repositorio.obtener(id);
    }
}

Ventajas

  • ✅ Interfaz consistente
  • ✅ Fácil cambiar implementación
  • ✅ Testeable (mock repository)
  • ✅ Lógica agnóstica a persistencia

Desventajas

  • ❌ Interfaz genérica puede ser limitante
  • ❌ Overhead de abstracción

⚙️ Unit of Work

Unit of Work es un patrón que gestiona transacciones y coordina múltiples cambios en objetos.

Problema

¿Cómo coordinar múltiples cambios en objetos y asegurar consistencia transaccional?

┌─────────────────────────────────────────────────┐ │ SIN Unit of Work (Inconsistencia) │ ├─────────────────────────────────────────────────┤ │ repositorio.guardar(envio1) ✓ │ │ repositorio.guardar(envio2) ✓ │ │ repositorio.guardar(envio3) ✗ Error │ │ │ │ Problema: envio1 y envio2 guardados, envio3 no │ │ BD inconsistente │ └─────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────┐ │ CON Unit of Work (Consistencia) │ ├─────────────────────────────────────────────────┤ │ unitOfWork.registrar(envio1) │ │ unitOfWork.registrar(envio2) │ │ unitOfWork.registrar(envio3) │ │ unitOfWork.commit() ← Todo o nada │ │ │ │ Ventaja: Transacción atómica │ └─────────────────────────────────────────────────┘

Solución

Crear un Unit of Work que gestione transacciones.

Paso 1: Unit of Work

public class UnitOfWork {
    private Connection conexion;
    private List nuevos = new ArrayList<>();
    private List modificados = new ArrayList<>();
    private List eliminados = new ArrayList<>();
    private RepositorioEnvio repositorio;
    
    public UnitOfWork(Connection conexion, RepositorioEnvio repositorio) {
        this.conexion = conexion;
        this.repositorio = repositorio;
    }
    
    // Registrar nuevo
    public void registrarNuevo(Envio envio) {
        nuevos.add(envio);
        System.out.println("✓ Registrado como nuevo: " + envio.getId());
    }
    
    // Registrar modificado
    public void registrarModificado(Envio envio) {
        modificados.add(envio);
        System.out.println("✓ Registrado como modificado: " + envio.getId());
    }
    
    // Registrar eliminado
    public void registrarEliminado(Envio envio) {
        eliminados.add(envio);
        System.out.println("✓ Registrado como eliminado: " + envio.getId());
    }
    
    // Commit (transacción)
    public void commit() {
        try {
            conexion.setAutoCommit(false);
            
            // Insertar nuevos
            for (Envio envio : nuevos) {
                repositorio.guardar(envio);
            }
            
            // Actualizar modificados
            for (Envio envio : modificados) {
                repositorio.actualizar(envio);
            }
            
            // Eliminar eliminados
            for (Envio envio : eliminados) {
                repositorio.eliminar(envio);
            }
            
            conexion.commit();
            System.out.println("✓ Transacción completada");
            limpiar();
        } catch (SQLException e) {
            try {
                conexion.rollback();
                System.out.println("✗ Transacción revertida: " + e.getMessage());
            } catch (SQLException ex) {
                ex.printStackTrace();
            }
        }
    }
    
    // Rollback
    public void rollback() {
        try {
            conexion.rollback();
            System.out.println("✓ Cambios revertidos");
            limpiar();
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }
    
    // Limpiar
    private void limpiar() {
        nuevos.clear();
        modificados.clear();
        eliminados.clear();
    }
}

Paso 2: Uso

// Crear Unit of Work
UnitOfWork unitOfWork = new UnitOfWork(conexion, repositorio);

// Registrar cambios
Envio envio1 = new Envio(1, "Buenos Aires", "Córdoba", 5.0);
Envio envio2 = new Envio(2, "Mendoza", "San Juan", 3.0);
Envio envio3 = new Envio(3, "La Plata", "Mar del Plata", 2.0);

unitOfWork.registrarNuevo(envio1);
unitOfWork.registrarNuevo(envio2);
unitOfWork.registrarNuevo(envio3);

// Commit (todo o nada)
unitOfWork.commit();

Ventajas

  • ✅ Consistencia transaccional
  • ✅ Rollback automático en errores
  • ✅ Agrupa múltiples cambios

Desventajas

  • ❌ Complejidad adicional
  • ❌ Requiere gestión manual

⏱️ Lazy Load

Lazy Load es un patrón que carga datos bajo demanda, mejorando rendimiento.

Problema

¿Cómo evitar cargar datos innecesarios que pueden no ser usados?

┌─────────────────────────────────────────────────┐ │ SIN Lazy Load (Carga Inmediata) │ ├─────────────────────────────────────────────────┤ │ Envio envio = repositorio.obtener(1) │ │ ├─ Carga envio │ │ ├─ Carga cliente │ │ ├─ Carga historial (1000 registros) │ │ └─ Carga documentos (100 archivos) │ │ │ │ Problema: Carga TODO aunque solo necesites ID │ └─────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────┐ │ CON Lazy Load (Carga Bajo Demanda) │ ├─────────────────────────────────────────────────┤ │ Envio envio = repositorio.obtener(1) │ │ ├─ Carga envio │ │ ├─ cliente = lazy (no cargado) │ │ ├─ historial = lazy (no cargado) │ │ └─ documentos = lazy (no cargado) │ │ │ │ if (necesitas cliente) │ │ cliente.obtenerNombre() ← Carga aquí │ │ │ │ Ventaja: Carga solo lo necesario │ └─────────────────────────────────────────────────┘

Solución

Crear un Proxy que carga datos bajo demanda.

Paso 1: Objeto Real

public class Cliente {
    private int id;
    private String nombre;
    private String email;
    
    public Cliente(int id, String nombre, String email) {
        this.id = id;
        this.nombre = nombre;
        this.email = email;
    }
    
    public String getNombre() { return nombre; }
    public String getEmail() { return email; }
}

Paso 2: Proxy Lazy Load

public class ClienteLazyProxy {
    private int id;
    private Cliente cliente;
    private RepositorioCliente repositorio;
    private boolean cargado = false;
    
    public ClienteLazyProxy(int id, RepositorioCliente repositorio) {
        this.id = id;
        this.repositorio = repositorio;
    }
    
    private void cargar() {
        if (!cargado) {
            this.cliente = repositorio.obtener(id);
            this.cargado = true;
            System.out.println("✓ Cliente cargado desde BD");
        }
    }
    
    public String getNombre() {
        cargar();
        return cliente.getNombre();
    }
    
    public String getEmail() {
        cargar();
        return cliente.getEmail();
    }
}

Paso 3: Uso

// Crear proxy (sin cargar)
ClienteLazyProxy cliente = new ClienteLazyProxy(1, repositorio);
System.out.println("Proxy creado, cliente NO cargado");

// Usar (carga aquí)
String nombre = cliente.getNombre();  // ← Carga desde BD
System.out.println("Nombre: " + nombre);

Ventajas

  • ✅ Mejora rendimiento
  • ✅ Carga bajo demanda
  • ✅ Transparente para cliente

Desventajas

  • ❌ Puede causar N+1 queries
  • ❌ Complejidad adicional

🏗️ Arquitectura Lógica

Cómo integrar todos los patrones en una arquitectura coherente.

Capas de una Arquitectura Típica

┌─────────────────────────────────────────────────┐ │ Capa de Presentación │ │ (Interfaz de Usuario) │ └─────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────┐ │ Capa de Aplicación │ │ (Servicios, Casos de Uso) │ └─────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────┐ │ Capa de Dominio │ │ (Lógica de Negocio, Entidades) │ └─────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────┐ │ Capa de Persistencia │ │ (Repository, Unit of Work, Data Mapper) │ └─────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────┐ │ Capa de Datos │ │ (Base de Datos) │ └─────────────────────────────────────────────────┘

Flujo de Interacción

1. Usuario solicita crear envío
   ↓
2. Controlador recibe solicitud
   ↓
3. Servicio (ServicioEnvios) procesa
   ↓
4. Servicio usa Repository
   ↓
5. Repository usa Unit of Work
   ↓
6. Unit of Work usa Data Mapper
   ↓
7. Data Mapper traduce a SQL
   ↓
8. SQL ejecuta en BD
   ↓
9. Resultado regresa por capas
   ↓
10. Usuario recibe respuesta

Ejemplo Completo

// 1. Capa de Presentación
public class ControladorEnvios {
    private ServicioEnvios servicio;
    
    public void crearEnvio(String origen, String destino, double peso) {
        servicio.procesarEnvio(origen, destino, peso);
    }
}

// 2. Capa de Aplicación
public class ServicioEnvios {
    private RepositorioEnvio repositorio;
    private UnitOfWork unitOfWork;
    
    public void procesarEnvio(String origen, String destino, double peso) {
        // Lógica de negocio
        Envio envio = new Envio(1, origen, destino, peso);
        
        // Persistencia
        unitOfWork.registrarNuevo(envio);
        unitOfWork.commit();
    }
}

// 3. Capa de Dominio
public class Envio {
    // Entidad de negocio pura
}

// 4. Capa de Persistencia
public class RepositorioEnvioSQL implements RepositorioEnvio {
    private EnvioMapper mapper;
    
    public void guardar(Envio envio) {
        mapper.insert(envio);
    }
}

// 5. Capa de Datos
// Base de datos SQL

Principios Clave

  • Separación de Responsabilidades: Cada capa tiene un propósito
  • Independencia: Cambios en BD no afectan lógica
  • Testabilidad: Cada capa puede testearse independientemente
  • Mantenibilidad: Código organizado y fácil de entender

📊 Comparación de Patrones

Patrón Propósito Complejidad Cuándo Usar
Data Mapper Mapear objetos a BD Media Cuando necesitas independencia objeto-BD
Repository Acceso uniforme a datos Media Cuando necesitas abstraer persistencia
Unit of Work Gestionar transacciones Alta Cuando necesitas consistencia transaccional
Lazy Load Cargar bajo demanda Media Cuando necesitas optimizar rendimiento

Matriz de Decisión

Escenario Patrón Recomendado
Aplicación pequeña, BD simple Repository
Aplicación empresarial, múltiples BD Data Mapper + Repository
Operaciones complejas con múltiples cambios Unit of Work
Datos grandes que pueden no usarse Lazy Load
Aplicación completa Todos combinados

💼 Casos Reales

Caso 1: Sistema de E-commerce

Requisitos: Múltiples tipos de productos, carrito de compras, historial de pedidos.

Patrones: Repository (para productos), Unit of Work (para carrito), Lazy Load (para historial).

Caso 2: Sistema Bancario

Requisitos: Transacciones atómicas, múltiples cuentas, auditoría.

Patrones: Unit of Work (transacciones), Data Mapper (múltiples tipos de cuenta), Repository (acceso uniforme).

Caso 3: Red Social

Requisitos: Millones de usuarios, posts, comentarios, notificaciones.

Patrones: Lazy Load (cargar comentarios bajo demanda), Repository (acceso uniforme), Data Mapper (mapeo complejo).

Caso 4: Sistema de Gestión de Logística (LogiSmart)

Requisitos: Envíos, centros de distribución, reportes, múltiples operaciones.

Patrones: Repository (acceso a envíos), Unit of Work (procesar múltiples envíos), Data Mapper (mapeo de entidades), Lazy Load (cargar detalles bajo demanda).

✅ Resumen

Conceptos Clave

  • Data Mapper: Traduce entre objetos y registros de BD
  • Repository: Proporciona interfaz uniforme para acceso a datos
  • Unit of Work: Gestiona transacciones y coordina cambios
  • Lazy Load: Carga datos bajo demanda para optimizar rendimiento

Arquitectura Lógica

  • Separar presentación, aplicación, dominio, persistencia y datos
  • Cada capa tiene responsabilidades claras
  • Cambios en una capa no afectan otras
  • Fácil de testear y mantener

Beneficios

  • ✅ Código más limpio y organizado
  • ✅ Fácil de testear
  • ✅ Fácil de cambiar BD
  • ✅ Mejor rendimiento
  • ✅ Mantenimiento más fácil
